System core

Start here. These four pages cover the philosophy, the token vocabulary, and the evidence basis behind every meaningful decision.

  • Preview Design System

    index.html

    Landing page and full component showcase. Live examples of every component, the token tour, and the hospital pressure-test row.

  • Principles

    docs/principles.html

    The six things that hold every page together: grayscale, tokens, plain HTML, accessibility, content-driven variants, and prefix discipline.

  • Research foundation

    docs/research.html

    Evidence basis with one citation per claim. WCAG 2.2 AA compliance map, audit history (v1.1 + v1.2), and the list of out-of-scope items.

  • Validation & testing

    docs/validation.html

    Protocol page covering Pa11y, Lighthouse, screen-reader audits, visual regression, browser/device/print matrices, and per-page-type performance budgets. Configs ship at the repo root.

  • Component status

    docs/component-status.html

    Every component tagged stable / beta / alpha / deprecated, with version-since metadata. Beta promotion happens after a documented WCAG audit pass.

  • Handoff

    HANDOFF.md

    Self-contained briefing for the next agent or contributor. File tree, architecture in 90 seconds, ten governance rules, ten lessons-learned gotchas, common task recipes, verification gates, and a quick reference.

  • Directory

    directory.html

    This page. Flat index of every HTML file in the system and every external reference it cites.

Foundations

The token layer and the layout utilities. Components reference these — nothing hardcodes.

  • Tokens

    styles/tokens.html

    Every token visualized — grayscale ramp with contrast ratios, type scale ladder, spacing strip, radius row, shadow stack.

  • Typography

    styles/typography.html

    Display, h1–h6, lead, body, small. Sans and serif. The drop-cap utility and the editorial moments.

  • Utilities

    docs/utilities.html

    Layout primitives (u-container, u-stack, u-cluster, u-grid, u-sidebar) and text helpers.

  • Page shells

    docs/page-shells.html

    Documented HTML scaffolds — landing, article, listing, detail. Each shell is markup, not a component class.

Component documentation

One page per component family. Each shows anatomy, variants, accessibility notes, and a code snippet.

  • Button

    components/buttons/

    Primary, secondary, ghost. Three sizes, icon-only, full-width, loading state. Tap targets clear 44px. Migrated to the new tabbed docs IA in v1.11.

  • Field

    components/field/

    The label + input + hint + error wrapper, with text input, textarea, and select. Help text and error states tied with aria-describedby.

  • Search

    components/search/

    A text input tuned for finding things — leading magnifier and the search input type.

  • Choice

    components/choice/

    Checkbox, radio, and toggle — pick any number, exactly one, or an instant on/off.

  • Form layout

    components/form-layout/

    The scaffolding that assembles fields — form, rows, and fieldset groups.

  • Breadcrumb

    components/breadcrumb/

    A trail showing the user's position in a hierarchy. The last item is the current page, not a link.

  • Pagination

    components/pagination/

    Previous/next and numbered pages for long, ordered result sets, with an ellipsis for the collapsed middle.

  • Top nav

    components/top-nav/

    The sticky masthead — brand, primary menu, utility actions, and an optional mega menu. Collapses to a burger + drawer on mobile.

  • Side nav

    components/side-nav/

    A vertical rail for local sections, in three patterns: peer pages, child pages, and a two-level section tree.

  • Section sub-nav

    components/subnav/

    A horizontal secondary nav band placed below the global masthead, scoping navigation to the current section or specialty. Five items max; is-current marks the active page.

  • In-page nav

    components/in-page-nav/

    A scroll-spy "on this page" table of contents that highlights the section in view on long pages.

  • Cards

    docs/cards.html

    Content, person, resource, stat. Bordered and elevated variants.

  • Hero

    components/sections/hero/

    Four variants: editorial (oversized type), split (text + media), centered, stat-led. Inverse modifier for dark surfaces.

  • Content blocks

    components/

    Feature grid, two-column, accordion (native <details>), tabs with arrow-key support, callout, pull quote, timeline.

  • Media

    components/media/

    Grayboxed image / video placeholders with ratio labels, plus the gallery (grid, mosaic) component.

  • Lists

    components/lists/

    Linked list, definition list, and article/news list patterns.

  • Citation list

    components/citation-list/

    Numbered references for bibliographies and evidence sections (beta).

  • Table

    components/table/

    Data tables and comparison tables. Striped and compact variants, semantic caption, scope on header cells.

  • Footer

    components/sections/footer/

    Compact, standard, expanded (with CTA row). Inverse modifier for dark surfaces.

Hospital demo — Riverside Medical Center

Nine representative templates built only from system classes and tokens. Zero inline styles, zero hardcoded values. If the system held here, it holds anywhere in healthcare.

  • Homepage

    pages/hospital-homepage.html

    Editorial hero, mega-menu IA, conditions grid, featured research, stats, news, expanded footer.

  • Specialty / condition landing

    pages/hospital-specialty.html

    Spinal cord injury page — split hero, what-we-treat list, care team person cards, related research resource cards, in-page nav.

  • Research & Innovation

    pages/hospital-research.html

    Stat-led hero, labs and teams two-column, publications list, timeline of milestones.

  • Find a doctor

    pages/hospital-find-a-doctor.html

    Listing page with sidebar filters (full form set), provider cards (person variant), pagination.

  • Provider profile

    pages/hospital-provider.html

    Dr. Maya Okonkwo — split hero, credentials deflist, tabs (About / Conditions / Research / Locations), stats, publication timeline.

  • Planning your visit

    pages/hospital-patient-visitor.html

    Long-form with in-page nav, FAQ accordion, text-size widget, help bar, print-friendly.

  • Location detail

    pages/hospital-location.html

    Riverside Streeterville — map placeholder, hours, accordion getting-here, services list.

  • Patient story

    pages/hospital-patient-story.html

    After his stroke, Marcus walked his daughter down the aisle — editorial typography, drop cap, pull quotes.

  • Research measure detail

    pages/hospital-measure.html

    Berg Balance Scale (BBS) — dense, structured-data archetype with deflist quick facts, two tables (population / psychometric), filterable citation list.

  • Appointment confirmation

    pages/hospital-appointment.html

    Multi-step booking flow exercising stepper, banner, modal, toast, skeleton, empty state, and tooltip in a single realistic layout.

  • Search results

    pages/hospital-search.html

    Filter sidebar (checkboxes, radios), active-filter chips, sorted article-list results with mark highlights, pagination. Empty-state demo below for the zero-result case.

  • Sign in

    pages/hospital-login.html

    Patient-portal sign-in. Reduced chrome, form-card layout, help bar with phone fallback, security callout.

  • Account settings

    pages/hospital-settings.html

    Portal preferences anchored by the v1.3 secondary-nav pattern. Form sections, notification matrix with toggles, accessibility controls (text-size widget), danger-zone modal.

  • 404 — Page not found

    pages/hospital-404.html

    Empty-state-heavy error page with recovery paths and a broken-link reporting affordance.

  • Patient portal — logged-in dashboard

    pages/hospital-portal.html

    Personalized landing for authenticated patients. At-a-glance stat cards (next visit, messages, results, balance), upcoming visit, recent test-result table, message preview list, goal empty state, quick-actions sidebar, care-team cards.

  • Appointment booking — pick a time

    pages/hospital-booking.html

    Step 2 of the booking flow: month-grid calendar with availability counts per day plus morning/afternoon time-slot grid. Stepper at top, filter sidebar (visit type, location, time of day), selection summary callout.

  • Medication detail — Baclofen

    pages/hospital-medication.html

    Patient-education drug archetype. Safety alert banner at top, split hero with quick-facts deflist, dose table, severity-tiered side-effect table, "when to call" callouts, related-medication linked list, pharmacy disclaimer.

  • Clinical trial — closed-loop SCI stimulation

    pages/hospital-trial.html

    Phase-2 trial detail with status sidebar, stepper-driven timeline, time-commitment table, interactive eligibility-check form (radios, checkboxes, toast-confirmed submit), risks/benefits, study-team cards, FAQ accordion.

Repo structure

The folder layout, for orientation. No build step — every file under css/ is plain CSS, every file under js/ is plain ES2017+ JavaScript.

/
├── index.html                  System landing & component showcase
├── directory.html              This page
├── README.md                   5-minute productive starter
├── .nojekyll                   Tell GitHub Pages to serve verbatim
│
├── css/
│   ├── wire.css                Entry — @imports the rest
│   ├── tokens.css              The only file with raw values
│   ├── reset.css
│   ├── base.css                Typography, links, focus rings
│   ├── layout.css              container, stack, cluster, grid, sidebar
│   ├── utilities.css
│   ├── print.css
│   └── components/             ~25 component partials, one file each
│
├── js/
│   └── wire.js                 Auto-init for tabs, accordion, drawer, etc.
│
├── docs/                       15 documentation pages
└── pages/                      9 hospital demos + 1 university demo

The token layer (css/tokens.css) is the one and only place raw values live. Components reference; nothing else hardcodes. Rebrand by editing tokens; the rest follows.
