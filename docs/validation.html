What we test, how we test it, and what we don't.

A research-backed system has to be a tested system. This page documents the protocol: which tools run on which pages, how often, what the budgets are, and — honestly — what hasn't been validated yet. Where a check requires tooling that doesn't ship with the repo, the runnable config is shipped alongside it.

Automated accessibility — Pa11y CI

Pa11y CI runs HTML CodeSniffer (WCAG 2.1 AA rules) against a static-served copy of the site. It catches the cheap mistakes — missing form labels, low contrast, duplicate IDs, bad ARIA — that a manual audit can miss when scope grows.

Config — .pa11yci.json (repo root)

Ships a list of every demo and docs URL under http://localhost:8080. Standard is WCAG2AA. Threshold is zero errors; warnings are allowed and logged for review.

Run

npx serve -l 8080 &
npx pa11y-ci

Expected baseline

Zero errors. Acceptable warnings: WCAG2AA.Principle1.Guideline1_3.1_3_1.H39.3.LayoutTable (false positives on the layout tables in hospital-portal.html and hospital-measure.html — both use caption + scope correctly).

Lighthouse CI

Lighthouse covers four scores per page: Performance, Accessibility, Best Practices, SEO. We hold Accessibility ≥ 95 and Best Practices ≥ 90 as hard floors; Performance and SEO are soft floors at 90.

Config — lighthouserc.json (repo root)

Configured against the same localhost URL set as Pa11y. Asserts the four floors. Per-page budgets (LCP, CLS, total bytes) documented below.

Run

npx serve -l 8080 &
npx @lhci/cli@latest autorun

Why we don't ship baseline scores yet

The scaffolding ships in this commit. The first run happens after a deploy target is wired up — likely Pages — so the numbers are taken against the actual network and host the rest of the world will see, not a localhost approximation.

Screen-reader audit

Automated tools catch about 30–40% of real accessibility issues (Deque, 2023). The rest take a human + a screen reader. Below is the protocol; the log lives at the bottom of this section and is updated after each pass.

Protocol

  1. NVDA on Windows 11 + Chrome — pass 1.
  2. VoiceOver on macOS + Safari — pass 2.
  3. For each pass: walk every page in the directory in order. Tab through every interactive element. Confirm landmarks announce, headings announce in order, links are descriptive, form labels associate, drawer / modal trap focus and restore on close, toast announces politely, text-size widget announces the scale change.
  4. Findings logged in this page's audit log section, with severity (critical / important / nice-to-have), the page and element, the AT, and the fix shipped (or planned).

Visual regression — Playwright snapshots

Catches accidental visual breaks across commits — a hero that broke at mobile width, a button color that drifted, a layout that collapsed when a token changed.

Plan

Playwright snapshot tests, one per demo page, at three widths (320, 768, 1280). Baselines stored under tests/snapshots/ and committed. CI fails on any diff above a pixel-difference threshold; developer review confirms intentional drift and updates the baseline.

Status

Not wired yet. Pre-conditions: a stable host, the print stylesheet's @page rules don't fight the snapshot viewport. Estimated effort: half a day.

How to run all of it locally

# One-time install
npm install

# Serve the static site
npm run serve            # http://localhost:8080

# Run validation in another terminal
npm run a11y             # Pa11y CI
npm run lh               # Lighthouse CI
npm run visual           # Playwright snapshots (after baselines captured)
npm run validate         # Runs all three in sequence

The package.json exposes the scripts and the dev-dependencies; nothing in the runtime ships. Add or remove pages from .pa11yci.json and lighthouserc.json as new ones land — the rest is wired.
