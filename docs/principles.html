System philosophy

Principles before pixels.

Six commitments the system makes to its users — designers, developers, and the clients they serve. If a component breaks one of these, the system is wrong, not the work.

1. Wireframes should provoke conversation about content, not color.

The system is entirely grayscale. Hierarchy comes from weight, scale, spacing, and contrast. Clients react to the structure on the page, not "what shade of blue is that." When color is decided later, the layout shouldn't have to change to accommodate it.

The grayscale ramp is designed against WCAG 2.2 AA contrast at every step. Body text always exceeds 4.5:1; large text always exceeds 3:1.

2. Names should be predictable, not clever.

Two prefixes, always:

.wire-*
Component class. Lives in a component file, follows BEM. Example: .wire-card, .wire-card__body, .wire-card--person, .wire-card.is-linked.
.u-*
Utility or layout primitive. Doesn't carry component identity; provides a single capability. Example: .u-stack, .u-text-muted.
data-wire-*
Behavior hook for the auto-init JS. Keeps behavior decoupled from styling. Example: data-wire-tabs, data-wire-drawer-open.
is-*
State, set by user or JS. Example: .is-current, .is-open, .is-loading.

Files and class names are lowercase, kebab-case. Always. No camelCase, no PascalCase, no abbreviations that aren't already in the system.

3. Variants are content variations, not style overrides.

A designer should be able to look at a component and pick the right variant without thinking. If they have to deliberate, the variant set is wrong.

What variants are: semantic decisions about emphasis (primary vs ghost), shape (icon-only), purpose (person vs resource).

What variants aren't: arbitrary style tweaks. No "extra-large with red border" buttons. If a one-off treatment is needed, that's a signal to either add a documented variant or push back on the design.

4. Accessibility is baked in, not bolted on.

  • Focus visible. Every interactive element shows a 2px focus ring with offset, regardless of background.
  • Touch targets. Buttons, links, and form controls are at least 44×44px.
  • Reduced motion. prefers-reduced-motion: reduce zeroes all transitions and animations.
  • Semantic HTML. Each component documents the elements and ARIA it expects. Wrap nav in <nav>, articles in <article>, etc.
  • Skip link. Every page shell includes a skip-to-content link.

5. The artifact should work everywhere, unchanged.

One folder, no build step. The same files serve:

  1. Local browsing — open index.html, click around.
  2. Figma Make — copy a component's HTML and the relevant CSS section.
  3. Claude artifacts — paste a flattened wire.css plus a page.
  4. Client handoff — zip the folder.
  5. Static review — host on GitHub Pages, Vercel, S3.

This is why there's no framework, no preprocessor, no bundler. Plain HTML, plain CSS, vanilla JS.

6. Extension is a documented pattern, not a free-for-all.

v1 does not ship a client-overlay system. When the second client lands, a single file like css/clients/<client>.css will override a constrained set of semantic tokens — never component selectors directly. The shape of that file is sketched here so we agree on the lane before paving it:

/* css/clients/example.css — illustrative only, not built yet */
:root {
  /* Allowed: re-aliasing semantic tokens to a different ramp step. */
  --color-accent: var(--gray-80);
  --color-bg-subtle: var(--gray-10);

  /* Allowed: tighter or looser scale, within reason. */
  --space-md: 1.125rem;
}

/* Allowed: client-scoped variants on existing components. */
.wire-button--example-pill { border-radius: var(--radius-pill); }

/* NOT allowed: overriding base component selectors. */
/* .wire-button { background: blue; } — no. */

If a client variant feels like it's straining the system, the answer is usually to talk to design — not to override the system.
