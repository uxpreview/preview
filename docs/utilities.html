Foundation

Utilities, primitives & layout

Small visual primitives (badges, tags, dividers) and single-purpose layout helpers that compose every page. Each one does one thing — label, group, separate, stack, cluster, grid, contain, hide — so components can stay focused on identity.

Visual primitives

Small, semantic UI atoms. They sit one level above tokens — close to components, but with no internal structure of their own.

Badge

An inline status label. Four variants: default, subtle, outline, inverse. An optional .wire-badge__dot precedes the label for "live" indicators.

When to use these primitives

  • Badge — short, persistent status (live, new, draft, accepting patients). Not for actions.
  • Tag — interactive: filter chips, active-filter pills, removable selections.
  • Divider — separate two unrelated sections within the same container. Skip it if a section heading already does the work.
  • Breadcrumb — deep pages (3+ levels). Skip on top-level pages and listings.
  • Pagination — paged listings of more than ~12 items. For infinite-scroll listings, use a "Load more" button instead.

When to reach for utilities

  • Use a utility when you need a single, one-off behavior that does not belong in a component.
  • Prefer layout primitives (.u-stack, .u-cluster, .u-grid) over margin utilities; gap-based spacing scales better.
  • Use .u-visually-hidden liberally for screen-reader labels — it is the cheapest accessibility win in the system.

When not to use a utility

  • Do not build a recurring pattern from a stack of utilities; once you reach for three or more, the pattern wants to be a component.
  • Do not chain margin utilities to recreate a stack — use the stack primitive.
