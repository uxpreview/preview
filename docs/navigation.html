Navigation

Top nav, mega menu, sidebar nav, in-page nav, breadcrumb, mobile drawer, and pagination. Each handles a different orienting question — where am I, where can I go, how do I get back.

Anatomy

A site typically uses a top nav across all pages, a breadcrumb on detail pages, in-page nav on long-form content, and a footer-anchored drawer for mobile. Pagination and sidebars appear on listing and section pages.

  • About
Make an appointment

When to use

  • Use top nav as the always-visible primary navigation across every page.
  • Use breadcrumb on pages two or more levels deep so users can step back without the back button.
  • Use in-page nav when a long article has three or more named sections.
  • Use sidebar nav on documentation, account, or settings pages with a stable secondary structure.
  • Use pagination when a list crosses one screen and infinite scroll would obscure totals.

When not to use

  • Do not duplicate the same links in top nav, breadcrumb, and sidebar — choose the primary surface for each.
  • Do not put a mega menu on a small site; a flat top nav is faster to scan.
  • Do not use breadcrumbs on a homepage or single-level site.

Accessibility

  • Wrap every navigation in <nav> with an aria-label distinguishing it (Primary, Breadcrumb, Pagination, On this page).
  • Mark the current item with .is-current and aria-current="page" (or "step", "location" as appropriate).
  • The mega menu opens with click and Escape closes it; keyboard focus moves into the panel on open.
  • Pagination uses aria-disabled on inactive prev/next; the current page is a non-link span.
  • Mobile drawer is wired by data-wire-drawer-open/data-wire-drawer-close; focus is trapped while open and restored on close.

Section navigation patterns

Three compositions of wire-sidenav for the three nav questions every healthcare and higher-ed site has to answer: where else can I go, what's inside this page, and where am I in the whole tree.

Pick the lightest pattern that answers the user's question. Local nav for peer pages, secondary nav for child pages, section tree only when the site IA is wide enough that the user can lose their place.

Local nav — peer pages

Flat list of sibling pages within the current section. The eyebrow at the top names the parent ("Conditions we treat") so the user knows what the peers are peers of.

Local nav (siblings)
Conditions we treat
  • Spinal cord injury
  • Stroke
  • Traumatic brain injury
  • Amputation
  • Multiple sclerosis
  • Parkinson disease

When to use

  • The current page sits inside a section of 4–12 peer pages and the user is likely to compare or move between them.
  • Condition / department / program / faculty-member listings where breadcrumb alone doesn't reveal the alternatives.

When not to use

  • Sections with more than ~15 peers — the rail becomes a scroll-jail. Use a search-driven listing page instead.
  • Single-page sections (one peer == no peers). Use a breadcrumb only.

Secondary nav — children of the current page

When a long-form topic is split into child pages (instead of in-page anchors), the parent link sits at the top of a wire-sidenav__sublist and each child is one nested entry below.

Secondary nav (children)
Planning your visit
  • Before your visit
  • Day of your visit
  • Parking & transit
  • What to bring
  • Visitor policies
  • After your visit

When to use

  • A topic large enough that splitting into 4–8 separate pages reads better than one long scroll.
  • Content where each child has independent value (deep linking, sharing, search ranking).

When not to use

  • Short topics. Prefer an wire-inpagenav anchor TOC on a single long page — less navigation overhead, same orientation.
  • Heavily-tabular comparisons where users want to scan multiple children side-by-side. Use wire-tabs.

Section tree — parent + siblings + children of current

Two levels of structure in one rail: the section parent at top, sibling items below, with the current item expanded to reveal its children. The richest pattern; reserve for sites where the IA is wide enough that users can lose their place.

When to use

  • Large sections where the user needs to see both their peers AND what's beneath them — Research with many programs, Academics with many schools, etc.
  • When the path the user took to get here is non-obvious (deep linking from search, social share).

When not to use

  • Small sections — adds noise without orientation value.
  • More than two levels of nesting. The system caps at parent → siblings → children of current. Deeper trees collapse to multi-page IA instead of trying to render them all.

Accessibility — all three patterns

  • Wrap each rail in a <nav aria-label="…"> with a label that describes the section ("Conditions we treat", "Research sections"). Pages that have more than one nav landmark must label each distinctly.
  • Set aria-current="page" on the current item's link. The is-current class provides the visual marker; the ARIA attribute is independent and required for assistive tech.
  • Touch targets meet WCAG 2.5.8 — wire-sidenav__link has min-block-size baked in. Sub-list links are smaller; verify ≥24px on any new addition.
  • Sticky behavior: rails inside wire-inpagenav are non-sticky below 48em per the v1.2 reflow fix. Plain wire-sidenav is not sticky by default; if you wrap it in a sticky container, repeat the breakpoint guard.
