Form design

Forms are where the real work happens — booking, applying, registering, paying. Make them short, legible, and forgiving. Every input has a label; every error is recoverable.

Structure

One column, top-to-bottom — multi-column forms invite skipped fields. Group related questions with a fieldset and lay them out with Form layout. The shortest form is the best form: ask only what you genuinely need right now, and defer the rest.

Labels and help

Every field has a visible, persistent label — a placeholder is not a label; it vanishes on input and fails contrast. (WCAG 2.2 — 3.3.2 Labels or Instructions, 1.3.1 Info and Relationships — requirements.)

  • Put help text and format hints before the field, not buried in a placeholder.
  • Mark optional fields rather than starring every required one, when most are required.
  • Use the Field component so label, hint, and error are wired together.

Validation and errors

Validate at the right moment — on submit or on blur, not on every keystroke — and make every error tell the user how to fix it. (WCAG 2.2 — 3.3.1 Error Identification, 3.3.3 Error Suggestion.)

  • On submit, show an error summary at the top that links to each bad field, and repeat the message inline.
  • Name the fix: "Enter a date in the future", not "Invalid date".
  • Preserve everything the user already typed — never clear the form on error.

Inputs sized for everyone

Controls are large and easy to hit — at least 44 px, with roomy tap areas (see Density). Use the right input type and inputmode so mobile keyboards match the task, and set autocomplete attributes so browsers can fill names, addresses, and phone numbers. (WCAG 2.2 — 1.3.5 Identify Input Purpose.)

Do
Keep the label, hint, and error visually connected to their field.
Don't
Use a placeholder as the only label, or disable submit to force a "complete" state.
