Toast

Beta

A small message that slides in to confirm something the system just did — saved, sent, copied — then leaves on its own. It never traps focus or blocks the page, so it is the lightest way to acknowledge an action. Screen readers hear it through a polite live region; errors get an assertive one.

  • css/components/toast.css
  • Molecule · requires js/wire.js

Live demo

Fire a toast and watch it slide in at the bottom-right, then auto-dismiss after five seconds. Hover it — or Tab to its close button — and the countdown pauses until you leave. The error fires through the assertive region instead of the polite one.

Overview

A toast is the gentlest acknowledgement in the system: it appears, says one short thing, and disappears. Because it never blocks, it is the right tool for confirming an action the user already took — not for anything they must read or act on. If the message can be missed without cost, a toast is right; if it can't, reach for something heavier.

When to use

  • Confirm an action the system completed on the user's behalf — "Saved", "Sent", "Link copied".
  • Acknowledge a background result that is good to know but not essential to see.
  • Give brief, low-stakes feedback that would be heavy as a banner and needless as a modal.

When not to use

  • For an error the user must fix — put it inline by the field, or in a banner. An assertive toast can announce an error, but it should not be the only place it lives.
  • For anything the user must acknowledge before continuing — that is a modal.
  • For persistent, page-level context (an outage, a policy change) — that is a banner.
  • In a flood. More than a couple at once is noise; batch them or switch to a banner.

Anatomy

A fixed region stacks toasts at the bottom-right; each toast pairs a content column (optional title + body) with a close button.

Types & variants

One shape, two announcement levels. The difference is which live region the toast renders into, not how it looks — staying grayscale, the assertive variant carries a heavier inline-start edge to read as serious.

VariantSelector / hookResult
Polite (default).wire-toastRenders into the role="status" / aria-live="polite" region. Announced when the reader is idle. For confirmations.
Assertive.wire-toast--assertive · data-toast-assertive / data-toast-variant="error"Renders into the role="alert" / aria-live="assertive" region, announced at once. Reserve for errors.

Behaviors

States

A toast is entering (added, then .is-open fades and slides it in), resting (visible, counting down), paused (pointer over it or focus inside — the countdown holds), or leaving (.is-open removed, then it is taken out of the DOM). For the visual treatment and tokens, see the Specs tab.

Interactions

Mouse

Hovering a toast pauses its auto-dismiss; leaving resumes it from where it stopped. Clicking the × dismisses it immediately.

Keyboard

The close button is a normal button in the tab order; moving focus into the toast also pauses the countdown, so a keyboard user is never raced to the close. Activating it dismisses the toast. The full contract is in the Accessibility tab.

Composability

  • Declarative trigger — any control with data-wire-toast-trigger and data-toast-title / data-toast-body / data-toast-duration. Add data-toast-assertive (or data-toast-variant="error") to route it to the assertive region.
  • Programmatic — call window.wireToast({ title, body, duration, assertive }) from your own code after an async result resolves.
  • Chaining — a modal's confirm button can both close the dialog and fire a toast, so the confirmation lands the moment the action commits.

In context

Confirmations firing after real actions in the demo flows.

  • hospital-appointment.html — a toast confirms the booking the moment the modal commits it.
  • hospital-settings.html — saving a preference acknowledges with a toast.
  • hospital-trial.html — the eligibility check reports back through a toast.

Do / Don't

Do

Confirm and get out of the way: "Appointment saved."

A done action acknowledged without stealing focus or a click.

Don't

Hide a required fix in a toast: "Payment failed."

If the user must act, it can't vanish — put it inline or in a banner.

Content guidelines

  • Lead with the outcome: "Saved", "Sent", "Copied". The title alone should carry the message.
  • Keep the body to one short sentence — a toast is glanced at, not read.
  • Don't put actions in a toast. If the user needs to click something, the message wants a banner or inline placement.
  • Write errors plainly and route them assertive — but never let the toast be the error's only home.

JavaScript

Toast requires js/wire.js: the live regions and every toast are created in script. There is no CSS-only fallback, so never let a toast be the only place a message lives.

  • window.wireToast({ title, body, duration, assertive }) — fires a toast. duration defaults to 5000ms; 0 disables auto-dismiss. assertive: true routes it to the assertive region.
  • data-wire-toast-trigger — on a control; reads data-toast-title, data-toast-body, data-toast-duration.
  • data-toast-assertive (or data-toast-variant="error") — routes the declarative toast to the assertive region.
  • Auto-dismiss pauses on mouseenter / focusin and resumes on mouseleave / focusout, counting the remaining time.

Code

<!-- Declarative: fire on click -->
<button class="wire-button"
        data-wire-toast-trigger
        data-toast-title="Appointment saved"
        data-toast-body="A confirmation is on its way.">Save</button>

<!-- Error, announced assertively -->
<button class="wire-button wire-button--secondary"
        data-wire-toast-trigger data-toast-variant="error"
        data-toast-title="Couldn't save changes">Try save</button>

<script src="js/wire.js" defer></script>

<!-- Programmatic -->
<script>
  wireToast({ title: "Link copied", duration: 3000 });
  wireToast({ title: "Upload failed", assertive: true });
</script>

ARIA & semantics

  • The polite region is role="status" + aria-live="polite"; the assertive region is role="alert" + aria-live="assertive". Both carry aria-atomic="false" so each new toast is announced on its own.
  • The regions exist before a toast is added, so assistive tech is watching when the content arrives (SC 4.1.3).
  • The close button has an explicit aria-label="Dismiss notification".
  • The toast carries no role of its own — the region's live semantics do the announcing.

Keyboard interactions

KeyAction
TabMoves to the close button in normal source order; doing so pauses the auto-dismiss countdown.
Enter / SpaceActivates the focused close button and dismisses the toast (native).

Toasts are not focus-stealing: nothing moves focus to them automatically, so a keyboard user is never yanked out of their task. They reach the close on their own terms, and the countdown waits while they do.

SCHow Toast satisfies it
1.4.1 Use of ColorEmphasis comes from the surface, weight, and the assertive edge — not hue. Grayscale throughout.
2.2.1 Timing AdjustableAuto-dismiss pauses on hover / focus and can be disabled (duration: 0). Critical content should not rely on it.
2.5.8 Target Size (Minimum) · AAThe × close target is 44 × 44 — well past the 24px floor.

Screen reader notes

  • Polite toasts are announced when the reader reaches a pause; assertive ones interrupt. Use assertive only for errors — interrupting for a routine "Saved" is rude.
  • Because the toast is removed after it leaves, the message is gone from the page; if the user needs to refer back to it, it wanted a banner.
  • Because the component is JS-dependent, confirm js/wire.js is present; without it nothing is announced.
  • Pending: a live NVDA + VoiceOver pass of both regions is not yet logged — this is why Toast is beta.
