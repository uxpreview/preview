Feature grid

Stable

A row of equally-weighted features or capabilities — each an icon, a title, and a short description — in two-, three-, or four-up grids, optionally bordered.

  • css/components/feature-grid.css
  • 4 widths

Live demo

A three-up grid of services — each cell pairs a small icon tile, a title, and one supporting sentence. The optional link sits at the foot of every cell.

Anatomy

The grid is a container plus repeated feature cells. Each cell stacks an optional icon, a title, a description, and an optional link. The width modifier on the container is the only thing that changes between layouts.

PartClassRole
Container.wire-feature-grid + widthThe grid; takes one width modifier (--2/--3/--4) and optional --bordered.
Cell.wire-featureOne feature — a stacked column of the parts below.
Icon.wire-feature__iconDecorative tile (aria-hidden="true"); a glyph or short token.
Title.wire-feature__titleThe feature name — a short phrase on a non-heading element.
Description.wire-feature__descOne supporting sentence in muted text.
Link.wire-feature__linkOptional action; pinned to the foot of the cell.

When to use

  • Use a feature grid for a set of equally-weighted items — services, capabilities, or value propositions — where no single one outranks the others.
  • Reach for three-up as the default; it reads comfortably on a laptop and collapses cleanly to two and then one column.
  • Use two-up when each item needs a longer description, and four-up for short, scannable labels.
  • Add --bordered when the features should feel like distinct, tappable tiles rather than a plain list.

When the items aren't peers — one promoted, the rest secondary — use a Two-column split or a set of Cards instead. A feature grid says "these are siblings."

Do / Don't

Do

  • Keep titles and descriptions roughly the same length so the cells read as equals.
  • Give every cell the same parts — don't link some and not others without reason.
  • Let the grid collapse to fewer columns on narrow screens; the cells reflow on their own.

Don't

  • Use a feature grid to rank items — equal weight is the whole point.
  • Lean on the icon to carry meaning; it's decorative scaffolding.
  • Pack four-up cells with long paragraphs; reach for two- or three-up instead.

Width reference

All grids start as a single column on small screens. The modifiers set how many columns appear once there's room; --bordered is orthogonal and pairs with any width.

WidthClassColumns
Two-up.wire-feature-grid--21 → 2 columns from 48em.
Three-up.wire-feature-grid--31 → 2 (48em) → 3 columns (64em).
Four-up.wire-feature-grid--41 → 2 (48em) → 4 columns (64em).
Bordered.wire-feature-grid--borderedModifier; gives each cell a border, padding, and equal height.

Tokens

PropertyToken
Grid gap--space-xl
Cell gap--space-sm

Bordered cell--space-xl padding, --border-thin --color-border, --radius-lg

Semantics

  • A feature grid is a set of peers — mark it up as a list (<ul> with <li> cells, or role="list") so assistive tech announces the count.
  • The icon is decorative scaffolding (aria-hidden="true"); the meaning lives in the title and description, never the glyph alone.
  • When a cell links, give the link a clear accessible name — "Learn more" alone is ambiguous in a list of links, so append the feature name (Learn more about inpatient rehab) with the extra words visually hidden.

Headings

  • Feature titles are short labels, not document headings — keep them on a non-heading element (<p> or <div>) so they don't fragment the page outline.
  • If the grid sits under a section heading, that heading carries the structure; the cells stay flat beneath it.

Responsive

  • Columns collapse to one on small screens and step up at 48em and 64em, so the reading order in the DOM is the order people hear and tab through.
  • The optional link is a full-size target pinned to the foot of each cell; keep its hit area at or above 44px (WCAG 2.2 SC 2.5.8 Target Size).
  • Hierarchy comes from scale, weight, and space — never color — so the grid reads the same in light and dark themes.
