Modal

Beta

A centered dialog that interrupts the page for a single decision or a short, focused task. Built to the ARIA dialog pattern: a scrim dims everything behind it, focus is trapped inside, Escape and the buttons close it, and focus returns to wherever it came from. Reserve it for moments worth blocking on.

  • css/components/modal.css
  • Organism · requires js/wire.js

Overview

A modal stops the page and asks for one thing: a confirmation, a decision, or a small piece of input. Because it traps focus and dims the rest of the screen, it is an interruption — powerful, and worth using sparingly. If the user can keep working around the message, it is not a modal; it is a banner or a toast.

When to use

  • A decision the user must resolve before continuing — confirm a deletion, accept terms, choose between two paths.
  • A short, self-contained task that benefits from focus — a quick edit, a single-purpose form.
  • A consequential action that deserves a deliberate, blocking confirmation (pair it with a static backdrop).

When not to use

  • For information the user can read and move past — use a banner or callout that doesn't trap them.
  • For a long form or a multi-step flow — give it its own page or a stepper; a modal that scrolls a lot defeats the focus it promises.
  • For content paning in from the side or a navigation surface — that is the drawer.
  • To nag. Every modal spends the user's attention; if it isn't worth blocking on, don't block.

Anatomy

A fixed overlay holds a dimmed backdrop and a centered panel. The panel stacks a header (title + close), a scrollable body, and a footer of actions.

PartClassRole
Root.wire-modalFixed, full-viewport overlay; role="dialog" + aria-modal set by JS. Carries id and aria-labelledby.

Types & variants

One structural type — a centered dialog — at two widths, with one behavior modifier. Width is a structural choice, not decoration.

VariantSelector / hookResult
Default.wire-modalPanel up to 32rem — confirmations and short messages.
Wide.wire-modal--lgPanel up to 48rem — short forms with a couple of fields.

Behaviors

States

A modal is closed (hidden, inert) or open (.is-open — the scrim fades in and the panel rises). There is no in-between; it is shown or it is not. For the visual treatment and tokens, see the Specs tab.

Interactions

Mouse

A control carrying data-wire-modal-open="id" opens the dialog. Clicking the backdrop closes it — unless the dialog is static — and so do the ×, and any footer button carrying data-wire-modal-close.

Keyboard

While open, Tab and Shift + Tab cycle through the dialog's controls and never leave it; Escape closes it; and on close, focus returns to the control that opened it. The full contract is in the Accessibility tab.

Focus & the page behind

Opening locks the body's scroll so the page behind can't move, and initial focus lands on the dialog's data-wire-modal-autofocus target — on a destructive confirm, the safe action — or the first control otherwise.

Composability

  • Trigger — any button with data-wire-modal-open="<modalId>". The modal markup can live anywhere on the page.
  • Title slot — .wire-modal__title with an id the root points at via aria-labelledby.
  • Body slot — any content: a sentence, a form, a short list. It scrolls independently if tall.
  • Footer slot — the action buttons. End with the primary action; put the safe / cancel action first and give it data-wire-modal-autofocus on destructive dialogs.
  • Chaining — a footer button can both close the modal and fire a toast (data-wire-toast-trigger) to confirm what happened, as the demos do.

In context

A confirmation step inside a real appointment flow, alongside a stepper, banner, toast, and skeleton.

  • hospital-appointment.html — the modal confirms a booking before it commits.
  • hospital-settings.html — a settings edit opens in a form modal.

Do / Don't

Do

Block on a real decision: "Cancel this appointment?"

A consequence the user should weigh before continuing — worth trapping focus for.

Don't

Interrupt to announce: "Your changes were saved."

Nothing to decide — a toast confirms it without stealing focus.

Content guidelines

  • Title the dialog with the decision, ideally as a question: "Cancel this appointment?" beats "Confirm".
  • State the consequence in the body plainly — what happens, and whether it can be undone.
  • Label buttons with the action, not "OK / Cancel": "Yes, cancel it" / "Keep appointment". The user should be able to act on the buttons alone.
  • Keep it short. If the body needs to scroll much, the content wants a page, not a modal.

Composition patterns

  • Confirm — a destructive or consequential action, static backdrop, safe action focused.
  • Quick edit — a short form in the wide variant, saved or discarded.
  • Acknowledge — terms or a notice the user must accept before proceeding.
  • Choose — a fork in a flow presented as two clear actions.

Related

  • Drawer — the side-anchored companion; same dialog mechanics, different geometry.
  • Toast — to confirm something without blocking.
  • Banner — for a persistent message the user can read past.
  • Stepper — when the task is a sequence, not a single decision.

JavaScript

Modal requires js/wire.js. Without it the dialog stays visibility: hidden and the trigger does nothing — there is no CSS-only fallback, so don't place essential content only inside a modal.

  • data-wire-modal-open="<modalId>" — on the trigger; initModal opens that dialog, locks body scroll, sets role="dialog" + aria-modal, and moves focus inside.
  • data-wire-modal-close — on the ×, the cancel button, or any control that should dismiss.
  • data-wire-modal-static — on the root; suppresses backdrop-click dismissal. Escape and close controls still work.
  • data-wire-modal-autofocus — on the control that should receive initial focus; falls back to the first focusable element.
  • aria-describedby is wired to the body automatically if the author hasn't set it.

Code

<button class="wire-button" data-wire-modal-open="confirm">Cancel appointment</button>

<div class="wire-modal" id="confirm" data-wire-modal-static aria-labelledby="confirm-title">
  <div class="wire-modal__backdrop"></div>
  <div class="wire-modal__panel">
    <header class="wire-modal__header">
      <h2 class="wire-modal__title" id="confirm-title">Cancel this appointment?</h2>
      <button class="wire-modal__close" data-wire-modal-close aria-label="Close">×</button>
    </header>
    <div class="wire-modal__body"><p>This releases your 10:30 AM slot.</p></div>
    <footer class="wire-modal__footer">
      <button class="wire-button wire-button--ghost" data-wire-modal-close data-wire-modal-autofocus>Keep appointment</button>
      <button class="wire-button" data-wire-modal-close>Yes, cancel it</button>
    </footer>
  </div>
</div>
<script src="js/wire.js" defer></script>

ARIA & semantics

The author names the dialog; initModal sets the dynamic dialog attributes on open.

  • role="dialog" and aria-modal="true" are set on the root, so assistive tech treats the panel as a modal and the page behind as inert.
  • aria-labelledby points at .wire-modal__title, giving the dialog its accessible name.
  • aria-describedby points at .wire-modal__body (wired automatically), giving the dialog its description.
  • The trigger gets aria-expanded reflecting the open state.

Focus management

  • Initial focus — moves into the dialog on open: to the data-wire-modal-autofocus control if present (the safe action on a destructive confirm), else the first focusable element.
  • Trap — Tab and Shift + Tab cycle within the dialog and never reach the page behind it, for as long as it is open.
  • Return — on close, focus returns to the control that opened the dialog, so keyboard users don't lose their place.
  • Scroll lock — the body's scroll is frozen while the dialog is open.

Keyboard interactions

KeyAction
Tab / Shift + TabMove to the next / previous control, wrapping within the dialog (focus trap).
EscCloses the dialog and returns focus to the trigger — even when the backdrop is static.
Enter / SpaceActivates the focused button (native).

A static backdrop suppresses only the click-outside-to-close shortcut. Escape and the explicit close / cancel controls always work, so the dialog is never a keyboard trap (SC 2.1.2).

Screen reader notes

  • On open, the dialog announces with its name (the title) and description (the body); aria-modal keeps the reader within it.
  • Because focus moves in, the user lands inside the dialog rather than being left behind it.
  • Because the component is JS-dependent, confirm js/wire.js is present; without it the dialog never opens.
  • Pending: a live NVDA + VoiceOver pass is not yet logged. Until it is, treat the screen-reader behavior as expected-but-unverified — this is why Modal is beta.
