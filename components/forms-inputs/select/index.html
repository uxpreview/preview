Select

Beta

A control for choosing one option from a fixed, predefined list, rendered inside a Field. Preview's Select is a styled native <select> by default — the most resilient choice for assistive tech, touch, and older users. It's the right tool for a handful of stable options, not the first reach for short or long lists.

  • css/components/form.css
  • Since v1.0

Live demo

A native select inside a field: a visible label bound to the control, and a non-value first option that prompts a choice without standing in for the label.

Single select

When to use

Select is the control of third choice for a single pick. Match the control to the number of options, and let input volume — not habit — decide.

OptionsUse
2–5, mutually exclusiveChoice (radio) — all options visible, lower cognitive load.
~6–15, stableSelect — constrains input and conserves vertical space.
16–100, or memorable (states, countries)Combobox (type-to-filter).
100+Search with typeahead.
More than one selectionCheckboxes or a multi-select listbox — not <select multiple>.

Public-sector research is openly cautious about selects — some users find them hard to operate — so treat the threshold as a ceiling, not a target. For yes/no or on/off, use a radio or toggle, not a Select. Don't use a Select for site navigation.

Anatomy

The label, hint, and error come from the Field wrapper; the Select itself is the trigger, the chevron, and the options.

PartClass / elementRole
Trigger.wire-select (native <select>)Shows the current value or the prompt; opens the option list.
Indicatorbuilt-in chevronSignals the control expands. Drawn in --color-text, no image asset.
Option<option>One choice; its text is its accessible name.
Group<optgroup label>Labels a category of related options.
Label / hint / error.wire-field__label · .wire-field__hint · .wire-field__errorProvided by the Field, not the control.

In context

Two moderate, known lists — one clinical, one academic. The university example groups options with <optgroup>.

Healthcare — appointment booking
Choose the option that best matches your appointment.
Higher-ed — program inquiry (grouped)

Do / Don't

Do

  • Keep a visible label above the field at all times.
  • Use radio (Choice) for a two-option yes/no answer.
  • Order options alphabetically, or by a clearer logical grouping.
  • Let people open, browse, and press Esc to keep their prior value.
  • Promote to a Combobox once a list runs past ~15 options.

Don't

  • Delete the label and lean on "Select a state" to name the field.
  • Put a yes/no or on/off choice in a Select.
  • List 50 states or 200 countries in raw database order.
  • Auto-submit the form the instant an option is chosen.
  • Use <select multiple> for multi-select.

States

Each state pairs a non-color cue with the change, so it reads in grayscale: a focus ring, a thicker border for errors, reduced contrast when disabled.

Rest · disabled · invalid
Select a reason for your visit.
StateGrayscale-safe cue
HoverBorder darkens to --color-text.
Focus-visibleVisible focus ring (never removed without a replacement).
DisabledSubtle fill, reduced text contrast, not-allowed cursor, out of focus order.
Invalid.is-invalid thickens the border; .wire-field__error adds an icon + message — never color alone.

Native vs. custom

Default to the native control. It submits reliably, invokes the OS picker on mobile, and needs no ARIA — which is why it's the most robust choice for an older, accessibility-sensitive audience.

Reach for a custom control only when grouping with bespoke styling, rich option content, or token-exact visuals genuinely require it — native options can't be richly styled. A custom Select must implement the ARIA APG Select-Only Combobox pattern (a combobox trigger with a listbox popup), the full keyboard model on the Accessibility tab, and grayscale state cues — then pass assistive-tech testing before release. "No ARIA is better than bad ARIA": a half-built custom widget is worse than the native element.

  </select>
</div>

Labeling

  • Every Select needs a visible, persistent label, bound with for/id via the Field. A placeholder option is a prompt, not a label — it must not replace one.
  • Mark required fields with the .wire-field__label-required modifier, never a bare asterisk.
  • Associate a hint or error with aria-describedby so it's announced with the control.

Semantics

  • Native: a labeled <select> needs no ARIA — the browser supplies role, state, and value. This is the default for good reason.
  • Custom: follow the APG Select-Only Combobox — a role="combobox" trigger with aria-expanded, aria-controls, and aria-activedescendant, plus a role="listbox" of role="option" children, the chosen one carrying aria-selected="true". DOM focus stays on the trigger.
  • The trigger border, chevron, and focus ring must meet 3:1 non-text contrast — in grayscale they carry all the meaning.

Keyboard

The browser provides this for native selects; a custom control must implement it in full (ARIA APG).

KeyAction
Down / Up / Enter / SpaceWhen closed, opens the list.
Down / UpWhen open, moves through options (stops at the ends).
Home / EndJumps to the first / last option.
Type a characterJumps to the next option starting with it (matching, not filtering).
Enter / SpaceCommits the focused option and closes.
TabCommits the value and moves focus onward.
EscCloses and keeps the prior value.

Targets, color & motion

  • Target size: the control is at least 44×44px — beyond the AA minimum, sized for older and motor-impaired users.
  • Use of color: open, focused, selected, and error states each pair a non-color cue (chevron rotation, focus ring, checkmark/weight, icon + thicker border). Color-only status fails WCAG 1.4.1.
  • Reduced motion: any chevron or popup animation honors prefers-reduced-motion.
