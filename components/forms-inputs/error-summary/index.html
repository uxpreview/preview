Error summary

Beta

When a form is submitted with errors, this box gathers them at the top — one link per problem, each jumping focus straight to the field that needs fixing. It's the reliable way to recover from a failed submit, especially on long forms and for screen-reader and keyboard users who'd otherwise have to hunt for what went wrong.

  • data-wire-error-summary

Overview

Inline errors tell you what's wrong once you reach a field; a summary tells you everything that's wrong at once, up front, and gets you there. Use both: the summary for orientation and recovery, the field error for the detail. The two must agree — each summary link repeats its field's message word for word, so there's no second guessing. Show the summary only when there are errors, place it at the very top of the form, and move focus to it the moment the submit fails.

When to use

  • Any form that can fail validation — especially longer ones where errors scroll off-screen.
  • Wherever recovering from a failed submit should be quick for keyboard and screen-reader users.

When not to use

  • For a single-field form — the inline field error alone is enough.
  • For success or neutral messages — that's a Banner or inline note, not an error summary.
  • Don't show it pre-emptively before the user has submitted.

References

Citations supporting this component, each labeled by evidence tier. Items marked unverified — pending review are drafts and must not be presented as established fact.

  • Convention — GOV.UK Design System: show an error summary at the top of the page on a failed submit, move focus to it, and link each entry to its field. Unverified — pending review. design-system.service.gov.uk
  • Standard — W3C WAI, WCAG 2.2 SC 3.3.1 Error Identification: errors are identified and described in text to the user. Unverified — pending review. w3.org
  • Standard — W3C WAI, WCAG 2.2 SC 3.3.3 Error Suggestion: each message says how to fix the problem ("…like name@example.com"). Unverified — pending review. w3.org
  • Standard — W3C WAI, WCAG 2.2 SC 2.4.3 Focus Order: moving focus to the summary, then to the chosen field, keeps a logical recovery order. Unverified — pending review. w3.org

AspectDetail

ARIA & semantics

  • Render the summary only when there are errors; on a failed submit, move focus to it (it takes tabindex="-1") so it's announced.
  • Give it an accessible name via aria-labelledby pointing at the title; the icon is aria-hidden.
  • Each link's text must match its field's inline error word for word, and point to the field's id.
  • Prefer moving focus over role="alert"; using both can double-announce.

Keyboard interactions

KeyAction
TabMoves through the error links in order.
EnterActivates a link and moves focus to the named field.

WCAG 2.2 criteria mapped

SCHow Error summary satisfies it
3.3.1 Error IdentificationLists each error in text, up front.
3.3.3 Error SuggestionMessages say how to fix the problem.
2.4.3 Focus OrderFocus moves to the summary, then to the chosen field.
1.4.1 Use of ColorSeverity from border + icon + text, not colour.

Screen reader notes

  • When focus moves to the summary, it reads "There's a problem" then the list of errors.
  • Activating a link announces the field's label and its inline error as focus lands.
