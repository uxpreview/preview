Textarea

Beta

A multi-line text input for free-form entry — a message to a care team, a reason for a visit, notes on a form. It's the native <textarea>, styled to match the single-line Field input and used inside the same field wrapper for its label, hint, and error.

  • css/components/form.css
  • Since v1.0 · no JavaScript

Live demo

A labelled textarea with a hint, and one in the invalid state. Each sits in a .wire-field so the label, hint, and error are wired to the control.

Overview

Reach for a textarea only when the answer is genuinely multi-line and open-ended. For a name, an email, or a date, a single-line Field input (or a more specific control) sets clearer expectations and is easier to complete. A textarea invites longer prose, so give it room and a label that tells the user what belongs there.

When to use

  • Free-form messages, descriptions, or notes that may run to several lines.
  • Anywhere the user can't predict the length and a single line would feel cramped.

When not to use

  • For short, structured values (name, phone, date) — use a single-line Field.
  • For choosing from known options — use Select or Choice.
  • For rich formatting — that's a rich-text editor, out of scope for the wireframe system.

Anatomy

A native <textarea> inside a Field, which supplies the label, optional hint, and error.

PartClass / elementRole
Wrapper.wire-fieldStacks label, control, and message; owns spacing.
Label.wire-field__label + forAlways-visible, programmatically tied to the control.
Control.wire-textarea on <textarea>The multi-line input. Vertically resizable.
Hint.wire-field__hint + aria-describedbyOptional guidance, announced with the field.
Error.wire-field__error + aria-describedbyShown with .is-invalid / aria-invalid.

States

StateSelector / attributeUse
Default.wire-textareaResting; shares the input border and focus ring.
Invalid.is-invalid + aria-invalid="true"Pairs with a .wire-field__error message.
DisableddisabledDimmed and non-interactive.
Sizingrows attributeSets the initial height; the user can resize vertically.

Content guidelines

  • Size the initial height (rows) to the expected answer, so a one-line reply doesn't sit in a huge box and a long one isn't cramped.
  • Use the hint for scope and limits ("Please don't include urgent information here"), not to restate the label.
  • If there's a character limit, state it in the hint and enforce it accessibly — never silently truncate.
  • Don't disable resizing; let users make room for their own text.

Tokens used

Textarea inherits the shared input rule (border, padding, radius, focus, disabled, invalid) and adds only its multi-line sizing. The relevant tokens:

TokenPurpose
--color-border / --color-border-strongResting and hover border.
--color-focusFocus ring (global :focus-visible).
--space-sm / --space-mdInner padding (block / inline).
--radius-smCorner radius, matching the input.

Measurements & structure

PropertyValue
Min heightmin-block-size: 7rem — a comfortable multi-line default.
Resizeresize: vertical — users can grow it, not break the layout horizontally.
Initial rowsSet per use via the rows attribute.
Shared baseBorder, focus, hover, disabled, and .is-invalid come from the .wire-input, .wire-textarea rule.

JavaScript

None. The native element handles input, resizing, and value. No data-wire-* hook. (Auto-grow on input would require script and is intentionally omitted.)

Code

<div class="wire-field">
  <label class="wire-field__label" for="msg">Message to your care team</label>
  <textarea class="wire-textarea" id="msg" rows="4" aria-describedby="msg-hint"></textarea>
  <p class="wire-field__hint" id="msg-hint">Please don't include urgent information here.</p>
</div>

<!-- Invalid -->
<div class="wire-field">
  <label class="wire-field__label" for="reason">Reason for visit</label>
  <textarea class="wire-textarea is-invalid" id="reason" rows="3"
            aria-invalid="true" aria-describedby="reason-err"></textarea>
  <p class="wire-field__error" id="reason-err">Please tell us the reason for your visit.</p>
</div>

ARIA & semantics

  • Use a native <textarea> with a real <label for> — that gives name and role with no ARIA.
  • Tie hint and error with aria-describedby so both are announced with the field.
  • Set aria-invalid="true" alongside .is-invalid so the error state isn't visual-only.
  • Never use a placeholder as the label; placeholders vanish on input and fail contrast.

Keyboard interactions

KeyAction
TabMoves focus to the textarea; the global focus ring appears.
EnterInserts a new line (unlike a single-line input, it does not submit).

WCAG 2.2 criteria mapped

SCHow Textarea satisfies it
3.3.2 Labels or InstructionsVisible label, plus an optional hint for scope/limits.
4.1.2 Name, Role, ValueNative element exposes name/role/value; aria-invalid carries the error state.
1.4.1 Use of ColorInvalid is shown by an icon + message, not colour alone.
2.4.7 Focus VisibleInherits the global focus ring.

Screen reader notes

  • Reads as "Message to your care team, edit text, multi-line" with its hint appended.
  • When invalid, it announces "invalid" and reads the associated error message.
