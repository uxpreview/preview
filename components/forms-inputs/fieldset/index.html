Fieldset

Beta

Groups related controls under a shared name. It's the native <fieldset> with a <legend>, which is what makes a set of radios or checkboxes — or a cluster of related fields — read as one question to assistive tech. Reach for it whenever several controls answer a single prompt.

  • css/components/form.css
  • Since v1.0 · no JavaScript

Overview

Some questions need more than one control: a set of radios for one choice, a group of checkboxes for several, or a few fields that belong together (a date split across three inputs, an address block). A fieldset ties them with a legend so a screen-reader user hears the group's prompt — "How should we contact you?" — before each option, instead of three orphaned labels. It's a semantic wrapper first; the visual grouping is light by design.

When to use

  • A radio group (one choice from several) — almost always wrap these.
  • A checkbox group answering one prompt.
  • A small cluster of fields that form one logical unit (date parts, an address).

When not to use

  • Around a single control — its own Field label is enough; a fieldset would be noise.
  • As a generic visual box — use a Card or layout, not form-grouping semantics.
  • To group unrelated fields just because they're adjacent.

PartClass / elementRole
Group.wire-fieldset on <fieldset>The grouping container; stacks its controls.

Do / Don't

Do

How should we contact you?

Wrap a radio group in a fieldset; the legend states the question once for everyone.

Don't

How should we contact you?

Don't use a plain heading over loose radios — the prompt isn't associated, so it's not announced with the choices.

Code

<fieldset class="wire-fieldset">
  <legend class="wire-fieldset__legend">How should we contact you?</legend>
  <label class="wire-radio"><input type="radio" name="contact" class="wire-radio__input" checked><span>Phone call</span></label>
  <label class="wire-radio"><input type="radio" name="contact" class="wire-radio__input"><span>Text message</span></label>
  <label class="wire-radio"><input type="radio" name="contact" class="wire-radio__input"><span>Email</span></label>
</fieldset>

ARIA & semantics

  • Use native <fieldset> + <legend> — it exposes the group and its name with no ARIA.
  • The legend must be the fieldset's first child to be the group's accessible name.
  • Always wrap a radio group; without grouping, the question isn't tied to the options.
  • Keep legends short — most screen readers repeat the legend with each control inside.

Keyboard interactions

A fieldset adds no keyboard behaviour of its own; focus and arrow-key navigation belong to the controls inside (e.g. arrow keys move within a radio group). The legend is not focusable.

WCAG 2.2 criteria mapped

SCHow Fieldset satisfies it
1.3.1 Info and RelationshipsVisual grouping is also programmatic via fieldset/legend.
3.3.2 Labels or InstructionsThe legend labels the whole group.
4.1.2 Name, Role, ValueThe group exposes its name (legend) to assistive tech.

Screen reader notes

  • Each control reads with the legend prepended: "How should we contact you? Phone call, radio button, 1 of 3".
  • This repetition is why legends should be concise — a long legend is read before every option.
