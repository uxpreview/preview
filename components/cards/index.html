Card

Stable

A bordered container that groups a small unit of related content into one tappable block — an article teaser, a clinician, a resource, a statistic. Four content types, three surfaces, and a horizontal layout, all composing on one base. Pure CSS: no script, equal height in a grid.

Overview

A card collects a few pieces of one thing — a heading, a line of description, maybe a face or a number — inside a shared border, so the eye reads them as a unit and the whole block becomes one target. Cards shine in a grid of peers: a row of clinicians, a set of articles, a panel of stats. The border and the spacing do the grouping; nothing here relies on color.

When to use

  • Content — a teaser for a long-form item (article, story, news), linking to the full page.
  • Person — staff, clinicians, researchers: a face, a name, a role.
  • Resource — a downloadable or watchable asset, with its type called out.
  • Stat — a single number landed prominently in a grid.

When not to use

  • When a linked list will do — long single-line entries don't need a card's frame.
  • For one isolated block with no peers — a card earns its border by sitting among others.
  • Don't mix card types in one grid; the silhouette break is jarring.

Anatomy

A container, an optional media slot, and a body of stacked parts. The footer pins to the bottom so it lines up across an equal-height row.

PartClassRole
Container.wire-cardThe bordered, rounded block; block-size: 100% for equal-height grids.
Media.wire-card__mediaOptional image slot, usually a .wire-media placeholder.
Body.wire-card__bodyThe padded stack; grows to fill the card.
Eyebrow.wire-card__eyebrowA small uppercase kicker above the title.
Title.wire-card__titleThe heading; link it for navigation.
Description.wire-card__descA line or two of supporting copy.
Meta.wire-card__meta · __meta-itemA row of small facts, dot-separated.
Footer.wire-card__footerPinned to the bottom; actions or a read-more.
Type parts__role · __credentials · __resource-type · __stat-value · __stat-label · __stat-trendThe slots specific to person / resource / stat.

Types & variants

Purpose modifiers come first (person, resource, stat); surface and layout modifiers stack on top. Content is the unmodified base — there is no --content.

ModifierKindResult
(none)TypeContent card — media, eyebrow, title, description, meta.
--personTypeSquare media, smaller title, role + credentials slots.
--resourceTypeA resource-type label and roomier body for an asset.
--statTypeA large value, a label, an optional trend; extra padding.
--subtleSurfaceMuted fill, no border.
--borderedSurfaceStronger border (clears 3:1 non-text contrast).
--inverseSurfaceInverse surface for a single high-emphasis card.
--linkedBehaviorThe whole card is a target: hover darkens the border, focus shows a ring.
--horizontalLayoutMedia beside the body for list rows; stacks on narrow screens.

Behaviors

States

A plain card is static. A --linked card adds hover (the border darkens to --color-text) and focus (the shared 2px ring). Selection and emphasis come from the border and surface, never color alone. For tokens, see the Specs tab.

Interactions

Mouse

On a linked card the whole block is clickable and the border responds on hover. On a multi-action card, only the title and explicit controls are clickable.

Keyboard

A linked card is a single focus stop that activates with Enter; a multi-action card exposes its title link and controls as separate stops. The right link pattern matters — see the Accessibility tab.

No JavaScript

Card is pure CSS. There's no script to load and nothing to initialize — it renders and behaves identically with JavaScript disabled.

Composability

  • One base, stacked modifiers — wire-card wire-card--person wire-card--bordered. A type, then a surface, then layout / behavior.
  • Optional slots — media, eyebrow, meta, and footer are all optional; include only what the content needs.
  • Equal height — in a u-grid, every card fills its row; a __footer pins to the bottom so footers align.
  • Type slots — person adds role + credentials, stat adds value + label + trend, resource adds a resource-type — don't mix slots across types.

In context

Cards carry most of the grids across the demo pages.

  • hospital-find-a-doctor.html — a directory of person cards.
  • hospital-homepage.html — content cards, stat cards, and resources on one page.
  • All components — this very index is a grid of linked cards.

Do / Don't

Do

Use one type per grid: a row of person cards.

A consistent silhouette lets the eye scan the set as peers.

Don't

Wrap a multi-link card in one big <a>.

Nesting links inside a card-link is invalid and traps assistive tech — link the title instead.

Content guidelines

  • Keep titles to one line or two; the card is a teaser, not the article.
  • Make the description earn its place — skip it rather than padding with filler.
  • Hold the meta row to two or three facts; it's a glance, not a record.
  • Front-load the stat value; the label explains the number, it doesn't compete with it.

Composition patterns

  • Directory grid — person cards in a u-grid--3.
  • Article teasers — content cards linking to long-form pages.
  • Stat band — a row of subtle stat cards under a hero.
  • Resource list — horizontal cards stacked as scannable rows.

Related

  • Linked list — for long single-line entries that don't need a frame.
  • Media — the placeholder that fills a card's media slot.
  • Feature grid — a layout of cards or feature cells.

JavaScript

None. Card is pure CSS — no data-wire-* hook, no initializer, no dependency on js/wire.js. It's safe to place essential content in a card; it renders the same with scripting off.

Code

<article class="wire-card">
  <div class="wire-card__media"><div class="wire-media" data-ratio="16:9"></div></div>
  <div class="wire-card__body">
    <span class="wire-card__eyebrow">Research</span>
    <h3 class="wire-card__title"><a href="/story">A new approach…</a></h3>
    <p class="wire-card__desc">One or two lines of teaser copy.</p>
    <div class="wire-card__meta">
      <span class="wire-card__meta-item">Apr 12, 2026</span>
      <span class="wire-card__meta-item">6 min read</span>
    </div>
  </div>
</article>

The link pattern

The single most important card decision — there are two valid shapes, chosen by whether the card has more than one action.

  • Single target, no inner links — make the whole card an <a class="wire-card wire-card--linked">. One link, one focus stop, the entire block clickable. This is how the component index works.
  • More than one action — keep the card an <article>, link the title only, and add --linked for the hover affordance. The title carries navigation; other controls stay separate.
  • Never wrap a card that contains links or buttons in an outer <a>. Nesting interactive elements is invalid HTML and breaks assistive tech.

Focus & keyboard

  • A whole-card link is a native <a> — focusable, Enter-activated, and showing the shared 2px focus ring (--linked:focus-visible).
  • A title-linked card exposes its title and any controls as ordinary focus stops in source order.
  • Don't add tabindex to make a non-link card focusable without giving it a real action; a focus stop with nothing to do is a trap for keyboard users.

Semantics & headings

  • Use <article> for a self-contained card; a grid of them is a list of articles.
  • Pick the title's heading level to fit the page outline — h3 under an h2 section, not a fixed level.
  • Person and content media are decorative placeholders here (aria-hidden); a real image needs alt — empty for decorative, descriptive for a meaningful photo.
  • The dot separators in the meta row are CSS ::before content, so they're not announced — the facts read cleanly.

WCAG 2.2 criteria mapped

SCHow Card satisfies it
1.4.1 Use of ColorGrouping and emphasis come from the border, surface, and spacing — never color. Grayscale throughout.
1.4.11 Non-text ContrastThe --bordered variant's border clears 3:1 against the page so the boundary is perceivable.
2.4.7 Focus VisibleA linked card shows the shared 2px ring at 2px offset.
4.1.2 Name, Role, ValueThe title link names the destination; a whole-card link is a single named <a>; no nested interactives.

Screen reader notes

  • A whole-card link announces once, with the card's text as its name — keep that text concise so the link name isn't a paragraph.
  • A title-linked card reads as an article with a link inside; surrounding text gives it context.
  • Decorative media is hidden, so a card announces its words, not "image, image".
  • In a grid, wrap the set in a list so a card's position ("3 of 9") is conveyed.
