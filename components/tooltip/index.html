Tooltip

Beta

A short text hint attached to a control, revealed on hover and on keyboard focus. It is the lightest way to add a sentence of context to a terse label or an icon-only button — plain text only, no interaction inside. Attach it with one attribute; the bubble is CSS, and a small script adds Escape-to-dismiss.

  • css/components/tooltip.css
  • Atom · CSS bubble; Esc-dismiss needs js/wire.js

Live demo

Hover or Tab to each control to reveal its hint. With focus on one, press Esc to dismiss it without losing your place — the hint returns the next time you focus or hover.

Overview

A tooltip buys a control one sentence of clarification without spending layout on it. It is supplementary by nature: the interface must still make sense if the hint never appears. Because it holds only plain text and nothing interactive, it stays inside what an on-hover hint can safely be.

When to use

  • Name an icon-only button — paired with an aria-label, never instead of one.
  • Add a brief clarification to a terse label or a value ("Time the form was last saved").
  • Surface a shortcut or unit that would clutter the control if shown inline.

When not to use

  • For anything interactive — a link or button inside the hint. That is a popover.
  • For long or essential content — if the user must read it to proceed, render it inline.
  • On a non-focusable element you expect keyboard users to reach — they can't focus it, so they'll never see the hint.
  • As the accessible name of a control on its own — pair it with real text or an aria-label.

Anatomy

There is no wrapper element. The tooltip is an attribute on the trigger; CSS draws the bubble as a pseudo-element from the attribute's text.

PartHookRole
Trigger[data-wire-tooltip="…"]Any element. The attribute value is the hint text.
Bubble::after pseudo-elementThe dark hint, drawn from content: attr(data-wire-tooltip). Not in the DOM as an element.
Placement[data-wire-tooltip-placement="…"]Optional: below, start, or end. Default is above.

Placement

PlacementHookResult
Above (default)—Bubble centered above the trigger.
Belowdata-wire-tooltip-placement="below"Bubble centered below — when there's no room above.
Start / Enddata-wire-tooltip-placement="start|end"Bubble to the inline-start or -end, vertically centered.

The bubble is a single non-wrapping line, so keep the text short. Long strings extend past the viewport rather than wrapping — that length wants inline text or a popover.

Behaviors

States

The bubble is hidden by default and shown while the trigger is hovered or focused. Pressing Esc moves it to dismissed (hidden while focus stays put); leaving or re-focusing the trigger clears that. For the visual treatment and tokens, see the Specs tab.

Interactions

Mouse

Hovering the trigger reveals the bubble; moving away hides it. There is no auto-close timer, so the hint stays as long as the pointer rests (1.4.13 Persistent).

Keyboard

Focusing the trigger reveals the bubble; Esc dismisses it without moving focus, and it returns on the next focus or hover. The full contract is in the Accessibility tab.

In context

  • hospital-appointment.html — icon-only actions and a "last saved" timestamp each carry a tooltip alongside their accessible name.

Do / Don't

Do

Clarify an icon button: aria-label "Share" + tooltip "Copy share link".

The label names it for everyone; the tooltip adds the detail on hover and focus.

Don't

Put a link in a tooltip: "See the full policy."

Interactive content can't live in a hover hint — use a popover or render it inline.

Related

  • Popover — for longer or interactive content that a tooltip can't hold.
  • Button — icon-only buttons are the most common tooltip host.
  • Field — for persistent help text, a field hint beats a tooltip.
  • All components — the full library.

JavaScript

The hint itself needs no JavaScript — hover and focus are pure CSS. js/wire.js adds only the dismissable leg of WCAG 1.4.13:

  • initTooltip — on Esc while a trigger is focused, it adds .is-tooltip-dismissed to hide the bubble without moving focus, and removes it on the next blur or mouseleave so the hint can return.
  • Without the script the bubble still appears on hover and focus; only Esc-to-dismiss is unavailable.

Code

<!-- Icon-only button: aria-label is the name, tooltip is the detail -->
<button class="wire-button wire-button--icon-only"
        aria-label="Share"
        data-wire-tooltip="Copy share link">
  <span aria-hidden="true">↗</span>
</button>

<!-- Placement variant -->
<button class="wire-button"
        data-wire-tooltip="Below the control"
        data-wire-tooltip-placement="below">Save</button>

<script src="js/wire.js" defer></script> <!-- adds Esc-to-dismiss -->

Content on hover or focus (1.4.13)

SC 1.4.13 sets three requirements for content shown on hover or focus. Here is how the tooltip meets each — and the honest caveat on the second.

RequirementHow it's met
DismissableEsc hides the bubble without moving focus (js/wire.js). This is the leg a CSS-only tooltip cannot satisfy on its own.
HoverableHovering the trigger keeps the bubble up; the bubble is pointer-events: none. Because there is a small gap, don't rely on dragging the pointer onto the bubble — keep hints short so this never matters.
PersistentNo auto-close timer — the bubble stays until hover / focus leaves or Esc dismisses it.

ARIA & semantics

  • The tooltip text is CSS-generated content, so it is not reliably exposed to assistive tech. Treat it as a visual supplement, never as the accessible name.
  • For an icon-only button, the aria-label is the accessible name; the tooltip merely repeats or extends it for sighted hover / focus users.
  • The trigger must be focusable for keyboard users to see the hint — use a real <button> or <a>, not a bare <span>, where keyboard reach matters.

Keyboard interactions

KeyAction
TabFocuses the trigger and reveals its hint.
EscDismisses the hint without moving focus; it returns on the next focus or hover.

WCAG 2.2 criteria mapped

SCHow Tooltip satisfies it
1.4.1 Use of ColorThe bubble is a grayscale dark surface; nothing relies on hue.
1.4.13 Content on Hover or FocusDismissable (Esc), hoverable, and persistent — see the contract above.
2.1.1 KeyboardFocusable triggers reveal the hint on focus, not hover alone.
4.1.2 Name, Role, ValueThe control's name comes from its text or aria-label, not the CSS hint.

Screen reader notes

  • Because the hint is CSS content, a screen reader may not announce it. Never put information a screen-reader user needs only in a tooltip.
  • Always give the host control a real accessible name (text or aria-label); the tooltip is the sighted-hover extra, not the source of truth.
  • If the detail is essential for everyone, render it as visible text or a field hint instead.
