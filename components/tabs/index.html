Tabs

Stable

Parallel content split into panels the user switches between, one at a time. Built to the ARIA tabs pattern with full keyboard support and optional URL-hash deep-linking. This very page runs on it.

Overview

Tabs divide one subject's content into peer sections and show a single panel at a time. They keep a long page short while keeping every section one click away. Reach for tabs when the sections are of similar importance and the user reads one, not all, at once.

When to use

  • Faceting one entity: a provider's Overview / Insurance / Reviews, or this page's Usage / Specs / Accessibility.
  • Switching a view of the same data, e.g. list versus calendar.
  • Keeping a small set (a handful) of peer sections compact.

When not to use

  • For an ordered process, use a Stepper; tabs imply free movement, not sequence.
  • For moving between pages, use a navigation component, not tabs.
  • When users must compare content across sections, show it together instead of hiding it behind tabs.
  • When the count grows large; too many tabs is a sign the content wants a different structure.

Anatomy

A container holds a tablist (the row of tabs) and one panel per tab. Each tab points at its panel; each panel back at its tab.

PartClassRole
Container.wire-tabsWraps the tablist and panels; carries data-wire-tabs (and optionally data-wire-tabs-hash).
Tablist.wire-tabs__listrole="tablist"; the row of tabs with the underline rail.
Tab.wire-tabs__tabA <button role="tab"> with aria-controls pointing at its panel.
Panel.wire-tabs__panelA role="tabpanel" shown only when its tab is selected (.is-active).

Types & variants

One structural type (horizontal tabs), with two visual treatments. There is no vertical variant; all current uses are horizontal.

VariantSelectorResult
Default.wire-tabsUnderlined tabs; the selected tab shows an accent underline and bolder label.
Pill.wire-tabs--pillTabs sit in a muted track; the selected tab is a filled pill with a soft shadow.
Sticky.wire-tabs--stickyPins the tablist below the header on scroll. Documentation chrome, defined in doc-shell.css.

Behaviors

States

A tab is default, hover (label darkens), or selected (accent underline, bolder label, its panel shown). Selection is conveyed by the underline and weight, not color alone. For the visual treatment and tokens, see the Specs tab.

Interactions

Mouse

Clicking a tab selects it and reveals its panel. The previously open panel is hidden.

Keyboard

Tab moves focus to the selected tab, then the arrow keys move between tabs and activate them (automatic activation, wrapping at the ends); Home and End jump to the first and last; Tab again moves into the panel. Full contract in the Accessibility tab.

Deep-linking

Add data-wire-tabs-hash and the active tab syncs to the URL hash via the History API, so a tab is linkable and survives the browser's back and forward buttons. This page uses it (note how the URL changes as you switch tabs above).

Composability

  • Tab + panel pairs — each tab carries a short label and is wired to exactly one panel by aria-controls / aria-labelledby and matching ids.
  • Panel content — any content: prose, tables, even another component. (This page's panels hold whole documentation layouts.)
  • Wiring — put data-wire-tabs on the container; add data-wire-tabs-hash for deep-linking. Include js/wire.js.

Do / Don't

Do

Peer sections of one subject: About · Insurance · Reviews.

The user dips into whichever section they need, in any order.

Don't

A sequence: Your info → Payment → Confirm.

Ordered steps belong in a stepper; tabs imply you can jump around freely.

Content guidelines

  • Keep labels to a word or two; they sit in a row and must stay scannable.
  • Make labels parallel in form ("About", "Insurance", "Reviews"), not a mix of nouns and verbs.
  • Front-load the distinguishing word; avoid a shared prefix on every tab.
  • Order tabs by expected use, with the most common first (it is selected by default).

Composition patterns

  • Entity detail — facets of a provider, program, or location page.
  • Component docs — Usage / Specs / Accessibility, deep-linked with data-wire-tabs-hash.
  • View switch — list versus calendar versus map of the same results.
  • Settings — grouping a small set of preference panels.

Related

  • Accordion — disclosure when sections can be open at once, or on very narrow screens.
  • Stepper — for an ordered, sequential flow.

JavaScript

Tabs require js/wire.js; without it no panel shows and the keyboard model is absent. This is the system's one JS-dependent disclosure (Accordion, by contrast, is native).

  • data-wire-tabs — initTabs manages the roving tabindex, aria-selected, and the active panel, and binds the arrow / Home / End keys with automatic activation.
  • data-wire-tabs-hash — syncs the active tab to the URL hash via history.replaceState, makes a tab deep-linkable, and listens for hashchange so back / forward work.
  • The author supplies the ARIA roles and ids; the script supplies the dynamic state.

Code

<div class="wire-tabs" data-wire-tabs>            <!-- + data-wire-tabs-hash to deep-link -->
  <div role="tablist" class="wire-tabs__list" aria-label="Appointment details">
    <button role="tab" id="t1" class="wire-tabs__tab"
            aria-controls="p1" aria-selected="true" tabindex="0">Overview</button>
    <button role="tab" id="t2" class="wire-tabs__tab"
            aria-controls="p2" aria-selected="false" tabindex="-1">Preparation</button>
  </div>
  <section role="tabpanel" id="p1" aria-labelledby="t1" class="wire-tabs__panel">…</section>
  <section role="tabpanel" id="p2" aria-labelledby="t2" class="wire-tabs__panel" hidden>…</section>
</div>
<script src="js/wire.js" defer></script>

ARIA & semantics

The author provides the roles and the wiring; initTabs manages the dynamic attributes.

  • role="tablist" on .wire-tabs__list, with an aria-label naming the set.
  • role="tab" on each tab (a native <button>), with an id, aria-controls pointing at its panel, and aria-selected.
  • role="tabpanel" on each panel, with an id, aria-labelledby pointing back at its tab, and hidden while inactive.
  • Roving tabindex: the selected tab is tabindex="0", the rest tabindex="-1", so Tab lands on one tab and arrows move within. initTabs maintains this.

Keyboard interactions

KeyAction
TabInto the tablist (lands on the selected tab); again, out to the active panel.
← / →Move to the previous / next tab and activate it, wrapping at the ends (automatic activation).
Home / EndMove to and activate the first / last tab.

Automatic activation is appropriate here because the panels are static and cheap to show. If a panel were expensive to load, the APG recommends manual activation (arrows move focus; Enter or Space activates).

Screen reader notes

  • A tab announces as "Overview, tab, selected, 1 of 3" (label, role, state, position).
  • The tablist is announced as a labeled group, so the set has context.
  • Activating a tab announces the new selection; the panel is reached via Tab and is associated by aria-labelledby.
  • Because the component is JS-dependent, confirm js/wire.js is present; without it a screen reader finds buttons with no panels.
