Button

Stable

A pill-shaped action with three emphasis levels and three sizes. Use exactly one primary per view; reach for ghost in dense interfaces.

  • css/components/button.css
  • Since v1.0

Overview

A button triggers an action — submitting a form, confirming a choice, advancing a flow. Emphasis communicates priority: there is one primary action per region, supported by secondary and ghost actions that recede.

When to use

  • Use primary for the single most important action in a view (one per region).
  • Use secondary for the supporting action paired with a primary.
  • Use ghost in toolbars, card footers, and dense surfaces where a bordered button would feel heavy.
  • Use icon-only for utility actions where the icon's meaning is unambiguous and an aria-label is provided.

When not to use

  • Do not use a button to navigate. Use an <a> styled as a button instead — see the Navigation family for link-vs-button guidance.
  • Do not stack three or more primary buttons; rework the page hierarchy.
  • Do not use icon-only buttons for destructive actions without confirmation.

Anatomy

A button is a single rounded element with an optional leading or trailing icon and a label. Default (44px) and large (52px) clear the 44px enhanced target; small is 36px, above the 24px AA minimum.

Types & variants

Button has a single type — there is one structural kind. Everything else is a variant: emphasis modifiers (primary, secondary, ghost), size modifiers (sm, lg), and shape modifiers (icon-only, full width) that compose on top.

Behaviors

States

Button has three states you set: default (ready), disabled (unavailable and non-interactive), and loading (the action is in progress, announced via aria-busy). Hover, focus, and active are interaction feedback, not states you author. For the visual treatment and tokens behind each, see the Specs tab.

default · disabled · loading

Interactions

Mouse

Clicking anywhere on the button activates it. The whole pill is the target; there is no separate hit area. On press it shifts down 1px for tactile feedback.

Keyboard

Tab moves focus to the button; Enter or Space activates it. For the full keyboard contract, see the Accessibility tab.

Composability

A button accepts a text label and, optionally, a leading or trailing icon. There are no named slots — content is ordered inline.

  • Label — required text in a <span> (or as the button's direct text).
  • Leading icon — a .wire-button__icon before the label; mark it aria-hidden="true".
  • Trailing icon — a .wire-button__icon after the label, same treatment.
  • Icon-only — drop the label, keep one .wire-button__icon, and supply an aria-label.
  • Grouping — cluster related buttons in a .wire-button-group.

Do / Don't

Do

One primary + one ghost. Cancel doesn't compete.

Don't

Three primaries means none of them is. Rework the hierarchy.

Content guidelines

Labels

  • Lead with a verb. "Save changes" beats "Changes".
  • Be specific. "Book appointment" beats "Submit".
  • Title case for short labels (1–3 words); sentence case for longer.
  • Avoid "Click here" — the action name is the affordance.

Emphasis pairing

A typical form footer has one primary, optionally one secondary, and a ghost cancel. See Typography for label type scale and casing.

Composition patterns

  • Hero CTAs — large primary + large ghost, side by side.
  • Card footers — small ghost, right-aligned.
  • Toolbars — small ghost icon-only, in a cluster.
  • Mobile form footers — full-width primary stacked above ghost cancel.

  • Standard — WCAG 2.2 SC 2.4.4 Link Purpose (In Context) (2023): purpose determinable from text/context; icon-only buttons require aria-label. Unverified — pending review. w3.org
  • Standard — WCAG 2.2 SC 2.4.7 Focus Visible (2023): :focus-visible applies a 2px solid focus ring at 2px offset on every interactive element. Unverified — pending review. w3.org
  • Standard — WCAG 2.2 SC 2.5.5 Target Size (Enhanced) (2023): AAA 44×44 CSS px; primary buttons meet this stricter floor. Unverified — pending review. w3.org
  • Standard — WCAG 2.2 SC 2.5.8 Target Size (Minimum) (2023): minimum 24×24 CSS px (AA); the system holds ≥44px on primary controls. Unverified — pending review. w3.org
  • Standard — WCAG 2.2 SC 4.1.3 Status Messages (2023): status changes programmatically determinable without moving focus; loading buttons accept aria-busy. Unverified — pending review. w3.org
  • Empirical — Nielsen Norman Group, Primary vs. Secondary Action Buttons (Pernice, 2015): a three-tier action hierarchy helps users identify the main action. Unverified — pending review (pre-2020 source). nngroup.com
  • Empirical — Nielsen Norman Group, Flat UI Elements Attract Less Attention and Cause Uncertainty: hover affordance via background fill, not underline-only. Unverified — pending review. nngroup.com
  • Convention — Laws of UX (Yablonski), Fitts's Law: time to acquire a target is a function of size and distance; reinforces ≥44px primary-control sizing. Unverified — pending review (agent-drafted candidate, not currently cited in research.html). lawsofux.com
  • Judgment — Preview Design System: pill button radius is an aesthetic carry-over from the MotherDuck reference; no evidence pills outperform rounded rectangles for usability. Design call, no external source claimed.

States

The full visual set, CSS-only and driven by the native element or a state class. What each state means lives in Usage › Behaviors.

StateTriggerWhat changes
Default—Resting fill, label, and border for the emphasis level.
Hover:hoverPrimary shifts to --color-accent-hover; secondary and ghost gain a --color-bg-muted fill (and secondary shifts its border). Runs at --duration-fast.
Focus:focus-visibleA 2px focus ring at 2px offset, always visible for keyboard users. Supplied by a shared global focus style, not button.css.
Active:activeA 1px translateY press depression.
Disabled:disabled / .is-disabledOpacity 0.4, cursor: not-allowed, pointer-events: none.
Loading.is-loading / [aria-busy="true"]Label color goes transparent, pointer-events: none, and a spinning ::after ring is drawn (disabled under prefers-reduced-motion). SR announces busy.

JavaScript

Button ships zero JavaScript of its own. button.css is pure CSS, and activation relies entirely on the native <button> element (Enter / Space, no script). There are no data-wire-* hooks for Button.

  • All states — :hover, :active, :disabled / .is-disabled, .is-loading / [aria-busy="true"], and the wire-spin keyframe spinner — are CSS-only.
  • Icon-only buttons need an aria-label; this is an authoring requirement, not a behavior.
  • The only data-wire-* attributes seen on button elements elsewhere (data-wire-drawer-open, data-wire-drawer-close) belong to the Drawer component being triggered, not to Button.

Code

<!-- Primary -->
<button class="wire-button" type="button">Confirm</button>

<!-- Secondary, small -->
<button class="wire-button wire-button--secondary wire-button--sm" type="button">
  Save for later
</button>

<!-- Ghost with leading icon -->
<button class="wire-button wire-button--ghost" type="button">
  <span class="wire-button__icon" aria-hidden="true">↓</span>
  <span>Download</span>
</button>

<!-- Icon-only -->
<button class="wire-button wire-button--icon-only" type="button" aria-label="Search">
  <span aria-hidden="true">⌕</span>
</button>

ARIA & semantics

  • Use a native <button type="button">. Never a <div> with a click handler.
  • Icon-only buttons MUST carry an aria-label describing the action.
  • Decorative icons inside labeled buttons use aria-hidden="true".
  • Loading state: set aria-busy="true"; assistive tech announces busy.
  • For form submission, use type="submit" explicitly.

Screen reader notes

  • NVDA + Firefox: announces "<label>, button". Icon-only announces the aria-label.
  • VoiceOver + Safari: same, plus "press Control-Option-Space to activate" hint.
  • Loading state (aria-busy="true") is announced as "busy" after the label.
  • Avoid changing the button label mid-activation; if state changes, use Toast or a live region rather than mutating the label.
