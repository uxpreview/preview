Drawer

Stable

Overview

A drawer is a panel that lives off-screen and slides in from an edge when summoned, dimming the page behind it. It is a dialog like the modal, but anchored to a side instead of centered — which makes it the natural home for a tall list: navigation, filters, a table of contents. Its most common use is the mobile menu, where the full nav can't fit across the top.

When to use

  • Mobile navigation: the primary menu, hidden behind the hamburger, slid in when needed.
  • A tall, secondary surface that benefits from the full height of the screen — filters, facets, a settings panel.
  • Anywhere a centered modal would be too small for the content but a full page would be too much.

When not to use

  • For a short decision or confirmation — that is a Modal, centered and small.
  • For primary navigation that fits on screen — show it; don't hide working nav behind a tap. Hidden navigation is discoverable only if the user thinks to open it.
  • For information the user can read past — use a banner or callout that doesn't trap focus.

Anatomy

A bare panel and its scrim. Unlike the modal, the drawer doesn't prescribe a header / body / footer — it holds whatever content you give it, usually a nav.

PartClassRole
Panel.wire-drawerThe off-canvas surface; role="dialog" + aria-modal applied by JS while open. Carries an id and an aria-label.
Backdrop.wire-drawer-backdropThe dimming scrim. A sibling that must immediately follow the panel in the DOM; clicking it closes the drawer.
Close.wire-drawer__closeThe dismiss control inside the panel, carrying data-wire-drawer-close.
Content—Any markup — typically a <nav> with a labelled list of links.

Types & variants

One panel, two anchors. The anchor is a structural choice about where the content comes from.

VariantSelectorResult
Default (end).wire-drawerAnchored to the inline-end; slides in from the right (in LTR).
Start.wire-drawer--startAnchored to the inline-start; slides in from the left. The direction the shell's own rail uses.

Width is fixed at min(22rem, 90vw) — wide enough for a menu, capped so it never covers the whole small screen.

Behaviors

States

A drawer is closed (translated off-screen, scrim hidden) or open (.is-open — it slides to its edge and the scrim fades in). For the visual treatment and tokens, see the Specs tab.

Interactions

Mouse

A control carrying data-wire-drawer-open="id" — usually the wire-topnav__burger — opens the panel. Clicking the backdrop closes it, and so does the close control (data-wire-drawer-close).

Keyboard

While open, Tab and Shift + Tab cycle within the panel and never leave it; Escape closes it; and on close, focus returns to the trigger. The full contract is in the Accessibility tab.

Landmark or dialog, by context

The script sets role="dialog" + aria-modal only while the panel is open, and strips them on close. That's what lets one element be a persistent navigation landmark at desktop width and a trapped, modal slide-over at mobile width — exactly how this site's left rail works. A drawer that is always off-canvas behaves as a dialog every time it opens; one that is a visible sidebar at desktop is just a landmark there, and only becomes a dialog when it slides over.

Composability

  • Trigger — any control with data-wire-drawer-open="<drawerId>"; on mobile this is the wire-topnav__burger.
  • Panel + backdrop — the .wire-drawer followed immediately by its .wire-drawer-backdrop sibling; the script pairs them by adjacency.
  • Content — a labelled <nav> is the common fill, but any content works: filters, a form, a TOC.
  • Close — a .wire-drawer__close control inside, plus Escape and the backdrop. Give the panel an aria-label so the dialog has a name.

In context

The clearest example is the page you're on: shrink the window below the rail breakpoint and the hamburger appears — that menu is a start-anchored drawer.

  • This site — the left rail collapses into a slide-over drawer on mobile, opened by the header burger.
  • hospital-homepage.html — the Riverside mobile menu is a drawer.

Do / Don't

Do

Hold a tall list: the full mobile menu.

A side panel uses the screen's height — room for navigation a top bar can't fit.

Don't

Pose a one-line decision: "Delete this?"

A short confirmation belongs in a centered modal, not a full-height panel.

Content guidelines

  • Put the close control first, where focus lands, so dismissing is the easiest thing to reach.
  • Label the panel (aria-label) and any nav inside it, so the dialog and its menu both announce with a name.
  • Keep the menu shallow; a drawer is for getting somewhere, not for deep browsing.
  • Mirror the anchor to the trigger's side when it helps — a left burger, a left drawer.

Composition patterns

  • Mobile menu — the primary nav behind the burger; the default pattern.
  • Persistent rail → slide-over — a desktop sidebar that becomes a drawer at mobile width (this site).
  • Filter panel — facets on a listing page, slid in from the start.
  • Detail peek — a side panel of detail without leaving the list.

JavaScript

Drawer requires js/wire.js. Without it the panel stays off-canvas and the trigger does nothing — so don't put unique content only in a drawer.

  • data-wire-drawer-open="<drawerId>" — on the trigger; initDrawer opens that panel, shows its backdrop, locks body scroll, and moves focus inside.
  • data-wire-drawer-close — on the close control (and the backdrop click and Escape do the same).
  • role="dialog" + aria-modal are added on open and removed on close, so a persistent sidebar isn't a dialog until it slides over. An existing role is preserved.
  • The backdrop is found as the panel's next sibling, so keep them adjacent.

Code

<button class="wire-button wire-button--icon-only" data-wire-drawer-open="menu" aria-label="Open menu">≡</button>

<aside class="wire-drawer wire-drawer--start" id="menu" aria-label="Main menu">
  <button class="wire-drawer__close" data-wire-drawer-close>Close</button>
  <nav aria-label="Main">
    <ul role="list">
      <li><a href="/find-a-doctor">Find a doctor</a></li>
      <li><a href="/locations">Locations</a></li>
    </ul>
  </nav>
</aside>
<div class="wire-drawer-backdrop"></div>     <!-- must follow the panel -->
<script src="js/wire.js" defer></script>

ARIA & semantics

The drawer's semantics shift with its state — the defining detail of the component.

  • While open, role="dialog" + aria-modal="true" are set, so the panel is a modal dialog and the page behind is treated as inert.
  • While closed (or persistent at desktop), those are removed — the element is a plain region / navigation landmark, not a dialog.
  • aria-label on the panel gives the dialog its name; label the inner <nav> too.
  • The trigger reflects state via aria-expanded and points at the panel with aria-controls.

Focus management

  • Initial focus — moves into the panel on open (the first focusable, typically the close control).
  • Trap — Tab and Shift + Tab cycle within the panel and never reach the page behind it while open.
  • Return — on close, focus returns to the trigger, so the user keeps their place.
  • Scroll lock — the body's scroll is frozen while open; the panel scrolls internally without chaining to the page.

Keyboard interactions

KeyAction
Tab / Shift + TabMove to the next / previous control, wrapping within the panel (focus trap).
EscCloses the panel and returns focus to the trigger.
Enter / SpaceActivates the focused link or button (native).

Screen reader notes

  • On open, the panel announces as a dialog with its name; aria-modal keeps the reader within it.
  • Because focus moves in, the user lands in the panel rather than behind it; closing returns them to the trigger.
  • When the same element is a persistent sidebar at desktop, it announces as a navigation region — not a dialog — because the dialog attributes are only present while it's slid over.
  • Because the component is JS-dependent, confirm js/wire.js is present; without it the panel never opens.
