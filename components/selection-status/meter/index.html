Meter

Beta

A scalar measurement within a known range — beds available, a satisfaction score, a fraction of a quota used. Built on the native <meter> element. Unlike Progress, it isn't moving toward done; it sits at a value, and the label tells you whether that value is good.

Overview

A meter shows where a known quantity sits within its range — full to empty, poor to excellent. It is a static reading, not motion toward completion; that is the line between it and Progress. The native element can mark a value as optimal, sub-optimal, or poor via its low/high/optimum attributes, which here render as grayscale density — the actual judgement always belongs in the text.

When to use

  • Capacity: beds available, seats left, quota used.
  • A score or rating against a fixed scale.
  • Any fraction of a whole that is measured, not progressing.

When not to use

  • For a task advancing over time, use Progress.
  • When "good vs bad" must be unmistakable, don't rely on the band shading — say it in words.
  • For a single headline number with no range, a KPI stat reads better.

Meter vs Progress

They look alike and are easy to confuse; the difference is what the value means.

QuestionMeterProgress
What is it?A measurement within a range.Completion of a task.
Does it move toward an end?No — it just reads where it is.Yes — toward 100%.
Example"3 of 20 beds free"."Upload 70% done".
Element<meter><progress>

Composability

A compact reading that drops into cards, rows, and dashboards.

  • In a location card — capacity at a glance ("beds available").
  • In a table row — a score column across items.
  • With a KPI stat — the stat is the headline, the meter shows it against its range.

Do / Don't

Do

Low · 3 of 20
3 of 20 beds

Put the reading and the judgement ("Low") in text. The bar shading only reinforces it.

Don't

A bare bar leans entirely on shading to say "this is low" — which grayscale can't, and a screen reader won't.

Content guidelines

  • Spell out the reading: "3 of 20", "4.2 / 5", "82% used".
  • If a value is good or bad, name it in words — don't leave it to the band shading.
  • State the unit and the whole, so the value has a frame ("of 20").
  • Keep low/high/optimum meaningful — they drive the density bands and the native semantics.

Related

  • Progress — completion over time, not a static reading.
  • KPI stat — a headline figure without a range bar.
  • Wait time — a specialised figure for current waits.

Measurements & structure

PropertyValue
Track height0.5rem default; 0.75rem under --lg.
FillDriven by native min/max/value — no inline styles.
Quality bandslow/high/optimum pick the band; the three map to grayscale density, never hue.
Styling hooks::-webkit-meter-bar, ::-webkit-meter-optimum-value / -suboptimum-value / -even-less-good-value, and ::-moz-meter-bar.
Value texttabular-nums so readings line up when stacked.

States

The "state" is the value within its range and which quality band it falls in (set by low/high/optimum). There are no hover/focus/active states — a meter is read, not operated.

JavaScript

None in the component. The application sets the attributes and keeps the value text in sync; the native element reflects the reading to assistive tech. No data-wire-* hooks.

Code

<!-- low/high/optimum pick the quality band; value places the fill -->
<div class="wire-meter">
  <div class="wire-meter__header">
    <label class="wire-meter__label" for="m1">ICU beds available</label>
    <span class="wire-meter__value">3 of 20</span>
  </div>
  <meter class="wire-meter__bar" id="m1"
         min="0" max="20" low="4" high="10" optimum="20" value="3">3 of 20 beds</meter>
</div>

<!-- "less is better" (e.g. quota used): set optimum low -->
<meter class="wire-meter__bar" min="0" max="100" high="80" optimum="0" value="82">82% used</meter>

ARIA & semantics

  • Use the native <meter>: it carries the meter role and exposes its value and range without extra ARIA.
  • Name it with a <label for>, and keep a text value reading beside it.
  • Don't rely on the quality band (optimum/sub-optimum/poor) to convey good-vs-bad — put that judgement in the value text.
  • Provide fallback text inside the element for browsers without <meter> support.

Keyboard interactions

None. A meter is an output, not a control — not focusable or operable. If users could change the value, that would be a slider.

WCAG 2.2 criteria mapped

SCHow Meter satisfies it
4.1.2 Name, Role, ValueThe native element supplies role and value; the label supplies the name.
1.4.1 Use of ColorThe reading and any good/bad judgement live in text; the grayscale bands only reinforce.
1.4.3 Contrast (Minimum)Label --color-text and value --color-text-muted on the surface clear 4.5:1.
1.4.11 Non-text ContrastThe bands are reinforcement, not the sole carrier of meaning, so the 3:1 graphic floor isn't load-bearing.

Screen reader notes

  • Support for <meter> varies; the visible text reading is what guarantees the value is conveyed.
  • The quality band is visual only — it is not reliably announced, so the judgement must be in the label or value text.
  • Keep the value text ("3 of 20") authoritative; treat the bar as a visual aid to it.
  • Provide fallback text inside <meter>…</meter> for non-supporting browsers.
