Progress

Beta

How far along a task is — a form being submitted, a file uploading, a multi-step flow. Built on the native <progress> element so the percentage rides on an attribute and the role and value are announced for free. Determinate when you know the share done; indeterminate when you only know it's working.

Overview

Progress answers "how much longer?" for a task in motion. It is about completion over time — distinct from Meter, which shows a static measurement within a range. Use the determinate form whenever you can estimate the share done; reserve indeterminate for genuinely unknown durations.

When to use

  • A file upload, a form submission, or a report being generated.
  • Step x of y in a wizard, as a thin completion bar.
  • Any wait long enough that silence would feel broken.

When not to use

  • For a fixed measurement (beds free, score out of 5), use Meter.
  • For a very short, unmeasurable wait, a Skeleton or spinner fits better.
  • Don't fake a determinate value you can't actually estimate — use indeterminate honestly.

Composability

A self-contained control that drops into flows and cards.

  • In a stepper — a thin completion bar beneath the step labels.
  • In a modal or card — while an action runs.
  • Vs Meter — progress moves toward done; meter sits at a measured value.

Content guidelines

  • Name what is progressing ("Uploading records"), not just "Loading".
  • Show the percentage as text alongside the bar for determinate progress.
  • For indeterminate, the label says it's working; don't imply a measurable share.
  • Round the value to whole percent — the bar isn't a precision instrument.

Related

  • Meter — a static measurement within a range, not motion toward done.
  • Stepper — named steps that a progress bar can summarise.
  • Skeleton — for short, unmeasurable content loads.
  • All components — the full library.

States

The meaningful "state" is the value itself, from 0 to max, plus the determinate/indeterminate distinction (presence of value). There are no hover/focus/active states — progress is read, not operated.

JavaScript

None in the component. The application updates the value attribute (and the visible percent text) as the task advances; the native element reflects the change to assistive tech. No data-wire-* hooks.

Code

<!-- Determinate: value/max drive fill AND the announced value -->
<div class="wire-progress">
  <div class="wire-progress__header">
    <label class="wire-progress__label" for="p1">Uploading records</label>
    <span class="wire-progress__value">70%</span>
  </div>
  <progress class="wire-progress__bar" id="p1" max="100" value="70">70%</progress>
</div>

<!-- Indeterminate: no value attribute; give it an aria-label -->
<progress class="wire-progress__bar" aria-label="Checking coverage, in progress"></progress>

<!-- Large track -->
<div class="wire-progress wire-progress--lg">…</div>

ARIA & semantics

  • Use the native <progress>: it carries role="progressbar" and exposes value/max with no ARIA needed.
  • Name it: a <label for> for determinate bars, or an aria-label for an indeterminate bar with no visible label.
  • Don't add a redundant role or aria-valuenow — the element already provides them.
  • Keep the visible percent text in sync with the value attribute; both should change together.

Keyboard interactions

None. Progress is an output, not a control — it isn't focusable or operable. (If users could set the value, that would be a slider, not progress.)

WCAG 2.2 criteria mapped

SCHow Progress satisfies it
4.1.2 Name, Role, ValueThe native element supplies role=progressbar and the current value; the label supplies the name.
4.1.3 Status MessagesValue updates are reflected by the element without stealing focus.
1.4.1 Use of ColorCompletion is in the value text and bar length, not hue. Grayscale guarantees no colour-only meaning.
2.3.3 Animation from InteractionsThe indeterminate animation is stilled under prefers-reduced-motion where the engine allows.

Screen reader notes

  • A determinate bar is announced with its name and percentage, e.g. "Uploading records, 70%".
  • An indeterminate bar is announced as busy; it has no percentage to read, so its label must say what's happening.
  • Because the element is native, value changes are picked up without an explicit live region in most screen readers.
  • The fallback text inside <progress>…</progress> shows in legacy browsers that don't support the element.
