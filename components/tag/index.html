Tag

Stable

A small chip with two jobs: an interactive control for filtering, selecting, and removing, and a read-only label for metadata. The element you render decides which.

Overview

A tag is a compact chip. It plays two roles, and which one you get is decided by the element you render it as:

  • Interactive — a <button> (toggle a filter, remove an item) or an <a> (navigate to a category). Focusable and operable.
  • Metadata — a <span>. A read-only label that shows an attribute. Not focusable, not clickable.

When to use

  • Filter sets where the user toggles facets on and off.
  • Applied filters or recipients the user can remove.
  • Read-only metadata on a card or row: specialties, languages, attributes.

When not to use

  • For a status the user only reads, use a Badge rather than a chip that looks clickable.
  • For the primary action on a view, use a Button.
  • Never render an interactive tag as a <span> with a click handler; it will not be reachable by keyboard.

Behaviors

States

An interactive tag has default, hover (a muted fill, CSS-only), and selected (a dark fill, toggled by a state class). A metadata tag is static. The visual treatment and tokens are in the Specs tab; the selected state must also be exposed with aria-pressed (see Accessibility), not by fill alone.

Interactions

Mouse

Clicking a filter tag toggles it on or off; clicking the × removes an applied tag. The component ships no JavaScript: it provides the look and the state class, and the application wires the toggle and the removal.

Keyboard

An interactive tag is a real <button> or <a>, so Tab reaches it and Enter or Space activates it. The remove × is a separate tab stop. Metadata tags are not in the tab order. Full contract in the Accessibility tab.

Composability

  • Label — the slot: a short word or two.
  • Remove — append a .wire-tag__remove for applied filters or removable input.
  • Group — wrap a set in .wire-tag-list (a <ul>) so they wrap and space evenly; label the list for context.

Do / Don't

Do

In-network

A button for the interactive filter, a span for the read-only label. The element matches the job.

Don't

Telehealth

Don't make a plain span interactive with a click handler. It is invisible to the keyboard and to screen readers.

Content guidelines

  • Keep labels to a word or two; a tag is not a sentence.
  • Use consistent casing across a set; title case for filters reads well.
  • Name filters as the facet, not an instruction: "Telehealth", not "Filter by telehealth".
  • For a removable tag, the remove button's label names what is removed: "Remove Telehealth".

Composition patterns

  • Filter bars — a .wire-tag-list of toggle buttons above a result set.
  • Applied filters — removable tags showing the active query, each with an ×.
  • Card and row metadata — --sm spans for specialties, languages, attributes.
  • Category links — anchor tags that navigate to a filtered listing.

Related

  • Badge — the non-interactive cousin, for status you only read.
  • Forms — checkboxes are the alternative when a filter set needs explicit form semantics.
  • Button — for primary actions rather than facets.
  • All components — the full library.

JavaScript

None ships. tag.css provides the look, the hover (CSS), and the selected visual (a state class). The application owns the behavior:

  • Selection — on click, toggle .is-selected (or --selected) and the matching aria-pressed value on the button.
  • Removal — on the × button, remove the tag element and move focus somewhere sensible (the next tag or the list).
  • There are no data-wire-* hooks for Tag.

Code

<!-- Filter set: interactive buttons, selection via aria-pressed -->
<ul class="wire-tag-list" aria-label="Filter by specialty">
  <li><button class="wire-tag wire-tag--selected" type="button" aria-pressed="true">Cardiology</button></li>
  <li><button class="wire-tag" type="button" aria-pressed="false">Neurology</button></li>
</ul>

<!-- Removable applied filter -->
<span class="wire-tag">Telehealth
  <button class="wire-tag__remove" type="button" aria-label="Remove Telehealth">
    <span aria-hidden="true">×</span>
  </button>
</span>

<!-- Read-only metadata (small) -->
<span class="wire-tag wire-tag--sm">In-network</span>

ARIA & semantics

  • Render an interactive tag as a real <button> (toggle or remove) or <a> (navigate). Never a <span> with a click handler.
  • Selection: set aria-pressed="true|false" on the toggle button. The fill is visual only; aria-pressed is what a screen reader reports.
  • Remove control: a <button> with aria-label="Remove <name>"; the × glyph is aria-hidden.
  • Metadata tag: a plain <span>, no role, not focusable.
  • Group a set in a <ul class="wire-tag-list"> with an aria-label describing the set.

Keyboard interactions

KeyAction
TabMoves focus to the next interactive tag, then to its remove button. Metadata tags are skipped.
Enter or SpaceToggles a filter, follows a link, or activates the remove button.

Screen reader notes

  • A filter button announces as "Cardiology, toggle button, pressed" or "not pressed" as aria-pressed changes.
  • A remove button announces "Remove Telehealth, button"; the × is not read because it is aria-hidden.
  • A metadata span reads as plain text inline; it is not announced as interactive.
  • The .wire-tag-list is read as a labeled list, so the set has context ("Filter by specialty, list, 4 items").
