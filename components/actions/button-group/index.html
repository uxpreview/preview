Button group

Beta

A thin container that lays out related buttons as one set — consistent gap, sensible wrapping, a clear reading order. It adds no button styling of its own; it groups buttons that already belong together, like Save / Cancel or a row of icon actions.

  • css/components/button.css
  • Molecule · no JavaScript

Live demo

Overview

When two or more buttons act on the same thing, grouping them signals that relationship and keeps spacing even. The group is deliberately minimal — a flex row with a token gap that wraps on narrow screens. All meaning still comes from the buttons inside it: one clear primary, the rest quieter.

When to use

  • Form or dialog actions — a primary plus one or two secondaries (Save / Cancel).
  • A small set of related icon buttons forming a toolbar.
  • A compact set of mutually related choices rendered as toggle buttons (aria-pressed).

When not to use

  • For many filter chips or removable selections — that is Tag, not a button group.
  • For a single-select control with many options — use a Select or radio Choice.
  • To stack more than one primary button — a group should still have a single most-important action.
  • For navigation links dressed as buttons — keep those as links; see the Navigation family.

Anatomy

One container, two or more buttons. The container owns only the gap and wrapping; it never restyles its children.

container · buttons
PartClassRole
Group.wire-button-groupThe flex container. Holds the gap and wrap behaviour; carries an optional role + label.
Buttons.wire-button (+ variants)Two or more related buttons, including icon buttons.

Composition patterns

The group has no style variants — the patterns come from the buttons you put in it.

PatternCompositionUse
Primary + dismissOne .wire-button + a .wire-button--ghostForm / dialog actions.
Icon toolbarSeveral .wire-button--icon-only, role="group" + labelA compact set of icon actions.
Toggle setButtons with aria-pressedA few related on/off choices.
Primary + dismiss · toggle set

Note — a visually attached / segmented group (buttons sharing borders with no gap) is not in the base today; this group is gap-separated. See the system gap noted in Code.

Do / Don't

Do

Save Cancel

Keep one clear primary and make the rest quieter. The group shows the actions belong together; emphasis shows which comes first.

Don't

Save Submit Send

Don't stack several primaries. Competing emphasis removes the hierarchy the group is meant to make obvious.

Content guidelines

  • Order by importance: the primary action comes first in reading order (and DOM order), the dismiss last.
  • Keep the set small — two or three actions. Move the rest into a Menu.
  • Label a toolbar group so its purpose is announced (e.g. "Text formatting").
  • Use parallel, specific verbs ("Save changes" / "Discard"), not vague pairs ("OK" / "No").

Related

  • Button and Icon button — what a group contains.
  • Menu — where extra or lower-priority actions belong.
  • Tag — for filter / removable selection sets, not action buttons.
  • Choice — radio/checkbox groups for single- or multi-select input.
  • All components — the full library.

References

Citations supporting this component, each labeled by evidence tier. Items marked unverified — pending review are drafts and must not be presented as established fact.

  • Convention — Gestalt principle of proximity: items placed close together are read as a related set, which is what the group's even gap signals. Unverified — pending review. lawsofux.com
  • Convention — Gestalt common region: a shared container reinforces that the buttons belong to one operation. Unverified — pending review. lawsofux.com
  • Empirical — Nielsen Norman Group: a clear primary-vs-secondary emphasis helps users choose the intended action; competing primaries blunt it. Unverified — pending review. nngroup.com
  • Standard — W3C WAI, WCAG 2.2 SC 4.1.2 Name, Role, Value: a toolbar set carries a group name (role="group"/"toolbar" + label) so its purpose is exposed. Unverified — pending review. w3.org
  • Standard — W3C WAI-ARIA APG, Toolbar pattern: a toolbar groups controls and manages focus with arrow keys (roving tabindex). Base ships no JS for this; see the system gap. Unverified — drafted, pending review. w3.org

Measurements & structure

PropertyValue
Displayinline-flex, so the group is only as wide as its buttons.
Gap--space-xs between each child.
Wrapflex-wrap: wrap — buttons drop to the next line rather than overflow or shrink.

States

The group has no states of its own. Hover, focus, active, disabled, loading, and pressed all belong to the individual buttons; the container only positions them.

JavaScript

None. The base group is static layout. A true ARIA toolbar with arrow-key roving focus would need a behaviour the system does not ship yet (no data-wire-* hook) — flagged as a system gap below. Until then, a group is keyboard-navigated as a normal sequence of buttons via Tab.

Code

<!-- Form actions: one primary, one ghost -->
<div class="wire-button-group">
  <button type="button" class="wire-button">Save changes</button>
  <button type="button" class="wire-button wire-button--ghost">Cancel</button>
</div>

<!-- Labelled icon toolbar -->
<div class="wire-button-group" role="group" aria-label="Text formatting">
  <button type="button" class="wire-button wire-button--secondary wire-button--icon-only" aria-pressed="true" aria-label="Bold">...</button>
  <button type="button" class="wire-button wire-button--secondary wire-button--icon-only" aria-pressed="false" aria-label="Italic">...</button>
</div>

System gap — there is no segmented / attached group variant (shared borders, no gap) and no roving-focus toolbar behaviour in the base. Both recur across products; noting them here so they can be promoted into the core deliberately rather than improvised per project.

ARIA & semantics

  • By default the container is presentational — each button is exposed on its own, which is correct for a Save / Cancel pair.
  • When the set is a deliberate toolbar (especially icon buttons), give it role="group" (or role="toolbar") and an aria-label so its purpose is announced.
  • For mutually exclusive choices rendered as toggles, carry the state on each button with aria-pressed; the group name describes the choice.
  • Don't add a group role with no label — an unnamed group adds noise without meaning.

Keyboard interactions

The base group is a normal tab sequence.

KeyAction
TabMoves to each button in turn; each shows its own focus ring.
Enter / SpaceActivates the focused button.
← →Toolbar only — roving focus between controls. Requires the toolbar behaviour the base does not yet ship (see Specs › JavaScript).

SCHow Button group satisfies it

2.5.8 Target Size (Min)Spacing comes from a real gap, so adjacent targets never collide; each button keeps its own ≥24px size.
1.3.1 Info & RelationshipsThe container groups related controls; reading and DOM order put the primary first.
1.4.1 Use of ColorPriority is shown by button emphasis and order, never hue. Grayscale guarantees it.

Screen reader notes

  • An unlabelled group reads as its individual buttons, in order — fine for a simple action pair.
  • A labelled toolbar announces the group name first ("Text formatting, group"), then each control.
  • Toggle buttons read their pressed state, e.g. "Bold, toggle button, pressed".
  • Putting the primary first means it is reached first by both tabbing and reading order.
