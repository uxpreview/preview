Menu

Beta

An action menu: a trigger button opens a popup list of commands. It follows the ARIA menu-button pattern — opening moves focus into the list, arrow keys roam it, Escape closes and returns focus. Use it to collect related actions behind one control, not for navigation or single-select input.

  • css/components/menu.css
  • Organism · data-wire-menu

Live demo

A labelled trigger and an icon-button overflow trigger. Open with click or ↓, roam with the arrow keys, choose with Enter, close with Esc.

Overview

A menu keeps a surface uncluttered by tucking secondary commands behind a single trigger. It is for actions — things that happen — arranged as a short list the user opens, scans, and dismisses. Because it overlays content and manages focus, it carries more weight than a button; reach for it only when a flat row of buttons would be too many.

When to use

  • An overflow ("⋯") of secondary actions on a card, row, or toolbar.
  • A small set (3–8) of related commands that don't all fit inline.
  • A grouped action list with a separator and the occasional destructive item.

When not to use

  • For navigation links — use a nav / mega menu; those are link lists, not action menus.
  • For choosing a value in a form — use a Select or Combobox (a listbox, not a menu).
  • For one or two actions — just show buttons or a Button group.
  • For a large, blocking decision — use a Modal.

Anatomy

A positioning root, a trigger button, and a popup list of menuitems. The behaviour hooks tie the three together.

PartClass / hookRole
Root.wire-menu + data-wire-menuRelative wrapper the popup positions against.
Triggerdata-wire-menu-trigger (a Button or Icon button)Toggles the menu; gets aria-haspopup/aria-expanded.
Popup.wire-menu__popup, role="menu", data-wire-menu-popupThe overlay list. Hidden until open.
Item.wire-menu__item, role="menuitem"One command. Use a real <button>.
Separator.wire-menu__separatorGroups items visually.
Group label.wire-menu__group-labelAn optional quiet heading for a group.

Types & variants

VariantSelectorUse
Start-aligned.wire-menu__popupDefault; aligns to the trigger's start edge.
End-aligned.wire-menu__popup--endAligns to the end edge — for a right-placed overflow.
Danger item.wire-menu__item--dangerA destructive command; emphasised by weight + icon, not colour.
Disabled itemaria-disabled="true"Present but not actionable; skipped by arrow keys.

Do / Don't

Do

Use role="menu" + role="menuitem" with real <button>s for a list of actions.

The roles and the wired keyboard model are what make it a menu, not just a floating box.

Don't

Don't put navigation <a> links inside a role="menu".

Links that navigate belong in a nav, not an action menu; mixing them confuses the announced role.

Related

  • Icon button — the usual overflow trigger.
  • Button group — when a few actions can sit inline instead.
  • Select / Combobox — for choosing a value (a listbox, not a menu).
  • Drawer and Modal — heavier overlays for larger tasks.
  • All components — the full library.

States

The popup is closed (hidden) or open (aria-expanded="true" on the trigger). Items show hover and :focus-visible with the same background wash, so the keyboard-focused item and the pointer-hovered item read identically. Disabled items (aria-disabled="true") are dimmed and skipped by the arrow keys.

JavaScript

Wired by js/wire.js via data-wire-menu on the root, data-wire-menu-trigger on the button, and data-wire-menu-popup on the list. The script manages aria-haspopup/aria-expanded, open/close, focus into and out of the list, arrow-key roaming, and outside-click / Escape / Tab-out dismissal. No application code needed beyond wiring each item's click to its action.

Code

<div class="wire-menu" data-wire-menu>
  <button type="button" class="wire-button wire-button--secondary" data-wire-menu-trigger>
    Actions
    <span class="wire-button__icon" aria-hidden="true"><svg ...>...</svg></span>
  </button>
  <ul class="wire-menu__popup" role="menu" aria-label="Actions" data-wire-menu-popup>
    <li role="none"><button type="button" role="menuitem" class="wire-menu__item">
      <span class="wire-menu__icon" aria-hidden="true"><svg ...>...</svg></span> Edit details
    </button></li>
    <li role="none"><hr class="wire-menu__separator"></li>
    <li role="none"><button type="button" role="menuitem" class="wire-menu__item wire-menu__item--danger">
      Cancel appointment
    </button></li>
  </ul>
</div>

<!-- Right-aligned overflow: icon trigger + end-aligned popup -->
<div class="wire-menu" data-wire-menu>
  <button type="button" class="wire-button wire-button--ghost wire-button--icon-only"
          data-wire-menu-trigger aria-label="More actions">...</button>
  <ul class="wire-menu__popup wire-menu__popup--end" role="menu" aria-label="More actions" data-wire-menu-popup>
    <li role="none"><button type="button" role="menuitem" class="wire-menu__item">Share</button></li>
  </ul>
</div>

ARIA & semantics

  • The trigger carries aria-haspopup="menu" and aria-expanded (managed for you); give the popup a role="menu" and an aria-label.
  • Each command is a <button role="menuitem"> wrapped in <li role="none"> so the list semantics don't double up.
  • Mark an unavailable command with aria-disabled="true" (keeps it discoverable but inert) rather than removing it.
  • This is an action menu — don't reuse the role for navigation links or for selecting a value (use a listbox/Select).

Keyboard interactions

KeyAction
Enter / SpaceOn the trigger: open the menu and focus the first item.
↓ / ↑On the trigger: open. In the menu: move to the next / previous item (wraps).
Home / EndJump to the first / last item.
EnterActivate the focused item, then close and return focus to the trigger.
EscClose the menu and return focus to the trigger.
TabClose the menu and continue past the trigger (no trap).

SCHow Menu satisfies it

2.1.1 KeyboardFully operable: open, roam, activate, and close from the keyboard.
2.1.2 No Keyboard TrapEscape and Tab always release focus from the menu.
2.4.3 Focus OrderOpening moves focus into the list; closing returns it to the trigger.

Screen reader notes

  • The trigger reads e.g. "Actions, menu, collapsed"; activating it announces the menu and the first item.
  • Items read as "Edit details, menu item, 1 of 3" as focus roams.
  • A disabled item reads as dimmed/unavailable and is skipped by the arrow keys.
  • Interactive menus are where SR behaviour diverges most across browsers — hence the pending audit noted under Usage.
