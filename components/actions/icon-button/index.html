Icon button

Beta

A compact, label-free button whose meaning rides entirely on a single icon. It is not a separate component — it is the icon-only application of Button (.wire-button--icon-only), so it keeps the same square ≥44px target, the same three emphasis levels, and the same states. The one rule it adds: because there are no visible words, it must carry an accessible name in text.

  • css/components/button.css
  • Atom · no JavaScript

Overview

An icon button trades a visible label for space. It earns its place only where the action is frequent, space-constrained, and represented by a near-universal icon — closing a dialog, opening an overflow menu, stepping a carousel. Everywhere else, a labelled Button is clearer, especially for the older and lower-vision audiences this system serves.

When to use

  • A toolbar or table-row action set where labels would not fit (edit, delete, more).
  • A universally understood control: close (×), search, print, share, expand.
  • The overflow trigger that opens a Menu of named actions.
  • A compact toggle — bookmark / save — paired with aria-pressed.

When not to use

  • The primary action of a page or form. That should be a labelled Button — never make the main call-to-action icon-only.
  • An action whose icon is ambiguous (a single icon meaning "export" vs "share" vs "upload"). If you would have to explain it, label it.
  • Navigation. A link that looks like an icon button is still a link — use an <a>, and see the Navigation family.
  • Anything where you cannot supply a real accessible name. No name, no icon button.

Anatomy

A square .wire-button--icon-only box centring one .wire-button__icon, plus an accessible name that lives in an attribute rather than on screen. The name is part of the anatomy, not an optional extra.

box · icon · (named) accessible label
PartClass / attributeRole
Box.wire-button.wire-button--icon-onlyThe square, pill-radius target. Carries emphasis, size, and state.
Icon.wire-button__iconA single <svg>, always decorative (aria-hidden, focusable="false").
Accessible namearia-label (or aria-labelledby)The verb the icon stands for. Required — it replaces the missing visible label.
Optional tooltipdata-wire-tooltipA visible hint on hover/focus. Complements the name; never replaces it.

Types & variants

Emphasis and size are inherited from Button unchanged — --icon-only only swaps the shape to a fixed square. Ghost is the most common for dense toolbars; primary is reserved for a single prominent action.

VariantSelectorUse
Primary.wire-button--icon-onlyOne prominent icon action; use sparingly.
Secondary.wire-button--secondary.wire-button--icon-onlyA bordered action that needs more weight than ghost.
Ghost.wire-button--ghost.wire-button--icon-onlyThe toolbar / row default — quiet until hovered.
Small.wire-button--icon-only.wire-button--sm2.25rem. Dense tables and inline rows.
Large.wire-button--icon-only.wire-button--lg3.25rem. Touch-first or prominent placements.
Togglearia-pressed="true|false"A two-state control (save, bookmark, mute).

Composability

The icon button is a connector. It rarely stands alone — it sits inside larger parts.

  • In a Button group — several icon buttons form a toolbar or segmented action set.
  • As a Menu trigger — the overflow "⋯" opens a list of named actions.
  • With a Tooltip — data-wire-tooltip adds a visible hint for the icon's meaning.
  • As a count host — wrap it in .wire-badge-anchor and pin a .wire-badge--count (e.g. notifications).
  • In a Card header or Modal — the close and overflow controls are icon buttons.

Do / Don't

Do

Give every icon button a specific aria-label ("Edit profile", not "Edit") and a matching tooltip. The name does the work the missing label would.

Don't

Don't ship a nameless icon button. A screen reader announces only "button", and the icon's meaning is lost to anyone who doesn't recognise the glyph.

Content guidelines

  • The accessible name is a verb phrase for the action, not a description of the icon — "Download results", not "Down arrow".
  • Make it specific when several siblings share a verb: "Edit profile" / "Edit address", so a screen-reader list of controls is unambiguous.
  • Keep the tooltip text and the aria-label in agreement, so sighted and screen-reader users hear the same thing.
  • For a toggle, name the action, not the state ("Save to my list"); the pressed state is carried by aria-pressed.

Tokens used

Identical to Button — the icon-only variant adds no tokens of its own, only fixes the box to a square. See Button's Specs for the full surface, hover, and focus token set.

TokenPurpose
--space-smInline padding the icon sits within (--btn-pad-x).
--radius-pillThe fully-rounded square corners.
--border-mdSecondary border and the loading spinner stroke.
Button's colour tokensForeground/background per emphasis — inherited unchanged from button.css.

Measurements & structure

The box is a fixed square so the icon is optically centred at every size. Like Button, these diameters are fixed component geometry in rem, not spacing tokens.

PropertyValue
Box — sm / md / lg2.25rem · 2.75rem · 3.25rem (square).
Icon.wire-button__icon at 1.25em; the SVG fills it 100%.
Shape--radius-pill; padding collapses to 0 with the fixed box.
Target floorThe default md (2.75rem ≈ 44px) meets AAA enhanced target size; sm (2.25rem ≈ 36px) still clears the 24px AA minimum.

States

All states are Button's, unchanged: hover background fill, a visible focus ring with offset, active press, disabled, and a loading state via aria-busy="true" (which swaps the icon for a spinner and announces busy). The only addition is aria-pressed for toggles.

default · disabled · loading · pressed

JavaScript

None of its own. The button is static markup. An optional Tooltip uses the CSS-only data-wire-tooltip attribute; js/wire.js only adds Escape-to-dismiss for it. Toggling aria-pressed and showing/hiding a busy spinner are the application's job.

Code

<!-- Ghost icon button — the toolbar default. Name is REQUIRED. -->
<button type="button" class="wire-button wire-button--ghost wire-button--icon-only" aria-label="Edit profile">
  <span class="wire-button__icon" aria-hidden="true"><svg ... focusable="false">...</svg></span>
</button>

<!-- With a visible tooltip hint (matches the aria-label) -->
<button type="button" class="wire-button wire-button--ghost wire-button--icon-only"
        data-wire-tooltip="Print" aria-label="Print lab results">
  <span class="wire-button__icon" aria-hidden="true"><svg ...>...</svg></span>
</button>

<!-- Sizes -->
<button class="wire-button wire-button--icon-only wire-button--sm" aria-label="Close">...</button>
<button class="wire-button wire-button--icon-only wire-button--lg" aria-label="Close">...</button>

<!-- Toggle: name the action, let aria-pressed carry the state -->
<button type="button" class="wire-button wire-button--ghost wire-button--icon-only"
        aria-pressed="false" aria-label="Save to my list">...</button>

<!-- Loading: aria-busy swaps the icon for a spinner and announces busy -->
<button type="button" class="wire-button wire-button--secondary wire-button--icon-only"
        aria-busy="true" aria-label="Saving">
  <span class="wire-button__icon" aria-hidden="true"><svg ...>...</svg></span>
  <span class="u-visually-hidden">Saving…</span>
</button>

<!-- As a count host (overlay from Badge) -->
<span class="wire-badge-anchor">
  <button class="wire-button wire-button--ghost wire-button--icon-only" aria-label="Notifications, 3 unread">...</button>
  <span class="wire-badge wire-badge--count" aria-hidden="true">3</span>
</span>

The accessible name

This is the whole reason an icon button is documented apart from Button. A labelled button gets its name from its text; an icon button has none, so you supply it.

  • Put the name on the <button> via aria-label="Edit profile", or point aria-labelledby at visible text elsewhere.
  • Mark the icon decorative — aria-hidden="true" on the wrapper and focusable="false" on the SVG — so it is never announced or focused.
  • A Tooltip (data-wire-tooltip) is a visible aid for sighted users. It is not guaranteed to be the accessible name, so always set aria-label too.
  • Avoid relying on the native title attribute as the name — its screen-reader and keyboard support is inconsistent.

ARIA & semantics

  • Use a real <button type="button">. Never an icon-button styled <div> or bare <svg> — you would lose role, focus, and keyboard for free.
  • For a two-state control, add aria-pressed="true|false"; keep the name the action ("Mute"), not the state.
  • For an overflow trigger that opens a Menu, pair it with aria-haspopup="menu" and aria-expanded.
  • The busy state uses aria-busy="true"; pair it with a visually-hidden status word so the change is announced (SC 4.1.3).

Keyboard interactions

Inherited from the native button, unchanged.

KeyAction
TabMoves focus to the button; a visible focus ring appears.
EnterActivates the button.
SpaceActivates the button.
EscDismisses a visible tooltip, if present (leaves focus in place).

Inside a Button group acting as a toolbar, arrow-key roving focus may apply — see that page.

Screen reader notes

  • A well-named button reads "Edit profile, button". A nameless one reads only "button" — the failure this page exists to prevent.
  • The decorative icon is skipped entirely, which is correct; the name carries the meaning.
  • A toggle reads its pressed state, e.g. "Save to my list, toggle button, pressed".
  • When several icon buttons share a verb, the per-item name ("Edit profile" vs "Edit address") keeps the rotor / elements list unambiguous.
